### Failure Mode Quality
- **Opus (10):** Three failure modes with deep structural analysis plus honorable mention. FM1 ("MFA skip on simple follow-ups") traces the compounding factor: check_name→step mapping is distributed across 700+ lines, so resuming at "earliest incomplete step" requires the LLM to maintain a mapping it has never seen as a single table. FM2 (5c materialization) decomposes all 6 phases and identifies the specific worst case: "passing unresolved `{staged_diff}` or `{panel_list}` tokens to a reviewer." FM3 (continue-immediately vs ask_user interleaving) is a novel observation — 4 progress signals interleaved with multiple conditional `ask_user` triggers create a "real pause vs fake pause" discrimination problem. Honorable mention for Investigation shortcut misclassification.
- **Sonnet (10):** Three failure modes with excellent mechanism analysis. FM3 (task sizing under-escalation for security-adjacent bugs) is novel — not raised by other models. - **GPT-5.4 (8):** Three valid failure modes. FM1 (MFA ordering conflict between "first runtime call" and Step 0) is valid. FM2 (continuation routing subjective) is correct. FM3 (5c complexity) is correct. Solid identification without the structural decomposition depth of Opus/Sonnet.
- **Codex (7):** Three correct failure modes (MFA bypass, 3a sequencing miss, 5c complexity overload). Correct identification but compressed — no structural decomposition, no compounding factors, no specific worst-case scenarios.

### Loop Start Diagnosis
- **Opus (10):** Three structural issues with deep analysis plus root cause assessment. Issue 1 (continuation logic swallows new tasks) with a concrete fix: "add a check like 'If the prior task has a commit row, it's complete — default to new task.'" Issue 2 (redundant loop-entry gate + potential infinite loop if LLM keeps failing at step 5). Issue 3 (Investigation shortcut short-circuits the loop — guard relies on LLM judgment at boost time, when it has the least context). Secondary cause: "sheer length of pre-code path (~25-35 tool calls) creates many opportunities to lose track." Correctly does NOT flag "no unconditional first tool call" (v0.10.6 fixed this).
- **Sonnet (10):** Three root causes with strong mechanism analysis plus a structural recommendation. RC1 (continuation misclassification suppresses start signal — prior task's `loop-entry` row persists, new request gets continuation-routed). RC2 (ambiguity gate fires before start signal emits — timing analysis shows start signal is deferred). RC3 (branch logic underspecified — "confident-but-wrong" continuation is the dangerous case). Recommendation: emit a continuation start signal (`Resuming at Step 4...`) to make loop-started state visible on every entry path. Novel and actionable. Correctly does NOT flag "no unconditional first tool call" (v0.10.6 fixed this).
- **GPT-5.4 (8):** Four structural causes: competing "first action" phrases, subjective continuation detection, start signal appearing only after sizing, distributed prerequisites. "Mostly defined but not fully unambiguous" — accurate assessment. Correct but thinner analysis.
- **Codex (8):** Four causes: step 5 skip → Step 0 gate blocks, continuation misclassification, early ambiguity/pushback pauses, runtime gate/tool-order violations. "Path is defined, but operationally fragile under instruction overload." Correct but compressed.

## Cross-Model Consensus

**Converging findings (3+ models):**
- `report_intent` vs `SELECT 1` ordering remains a subtle ambiguity — all 4 models note the "is report_intent a runtime tool call?" question. v0.10.6 added the rule but the distinction between UI label and runtime call is non-obvious.
- Step 5c prompt materialization remains the most procedurally complex single step — 3 models explicitly decompose phase-level breakpoints. Opus/Sonnet identify specific worst-case scenarios (unresolved placeholders, false HALTs from code variables).
- Continuation misclassification is now the primary loop-start risk — all 4 models flag it. This is a judgment call with no objective operationalization. The scope checks ("same files, same feature, same bug") are inherently fuzzy.

**Novel per-model insights:**
- **Opus:** Forward references in MFA (CREATE TABLE SQL is NOT inlined, forces 400+ line lookahead). "Continue immediately" vs `ask_user` interleaving creates a real/fake pause discrimination problem across 4 progress signals. Fix: add a "task complete" check (`check_name = 'commit'` row exists → new task).
- **Sonnet:** Continuation turns suppress the start signal entirely — user sees implementation without the 🔁 banner. Fix: emit a "Resuming at Step N" signal on continuation paths. Task sizing under-escalation for security-adjacent code — sizes by request description instead of codebase risk profile.
- **GPT-5.4:** Competing "first action" phrases across sections create confusion about what's truly first.
- **Codex:** `report_intent` dependency ordering can be mishandled by models not disciplined about tool call sequencing.

**What v0.10.6 fixed (confirmed by absence):**
- **Identity text before MFA** — flagged by Opus/Sonnet in v0.10.5 ("3 paragraphs of identity precede MFA"), no longer raised by any model. MFA-first positioning is working.
- **"No unconditional first tool call is mandated"** — flagged by Sonnet in v0.10.5, now resolved. All 4 models acknowledge the hard `SELECT 1` anchor. Remaining concern is the `report_intent` sequencing edge case, not the absence of a mandate.
