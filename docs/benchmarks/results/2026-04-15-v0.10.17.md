## Changes under test (v0.10.17)

- **Investigation continuation fix**: MFA Step C now splits `task-complete` (hard close — always new task) from `investigation-complete` (soft close — evaluate scope and mode before deciding). Follow-up messages during an investigation no longer trigger full MFA re-entry.
- **Mode-change guard**: If a follow-up after an investigation requests code changes ("fix it", "add logging"), the soft close detects the mode shift and forces a new code-change task instead of continuing as investigation.
- **Continuations section split**: Separate guidance for code-change continuations (full loop) vs investigation continuations (Survey → Present → investigation-complete → stop).
- **Task Sizing `ask_user` scoping**: Investigation classification confirmation via `ask_user` now scoped to new tasks only — not re-confirmed on in-scope investigation continuations.

## Notable Findings

### Opus — 50/50 (perfect)

- Identified 32 exact pre-code actions with specific SQL/bash commands — the most granular sequence of any model.
- Found all 9 distinct gates (11 evaluations) and correctly distinguished SQL-verifiable vs conversation-state checks.
- Top failure mode: **Step C is a decision tree maze** with 4-level nested conditionals, 3 sequential SQL queries, and the "discard the task_id" instruction appearing 3 times with different triggers. The investigation continuation logic adds another branch but Opus tracked it cleanly.
- Identified Step E's retry loop with no max-retry count as a structural gap — a new finding not seen in v0.10.15.
- Start signal should move earlier (immediately after loop-entry verification) — echoes Sonnet's v0.10.15 recommendation.

### Sonnet — 46/50

- Comprehensive sequence and gate identification, on par with Opus for Q1/Q2.
- Top failure mode: **MFA skip for "small" changes** — structural observation that nothing at the tool level enforces MFA before `edit`/`create`; enforcement is purely self-check instruction. Unique angle vs other models.
- Step C continuation routing correctly analyzed with the new investigation-complete soft close branch. Identified the stale `loop-entry` row + false-positive continuation as the highest structural fragility.
- MFA clarity rated 7/10 — lower than Opus's 8/10. Justification is solid (Step C complexity, batching ambiguity) but less structural than Opus's analysis. The -2 vs baseline is concentrated in Q3 (failure mode depth slightly less sharp) and Q4 (MFA clarity rating lower with less structural reasoning).
- Architectural recommendation: start signal should fire immediately after loop-entry, before Step 0 work.

### GPT-5.4 — 42/50

- 22-step sequence in correct order. Correctly picked up the split `task-complete` / `investigation-complete` check in Step C.
- Gate table includes all scored gates plus "size-escalation stop" (not a scored 🚫 GATE, but shows thoroughness).
- Failure mode #3 — "some gates are not mechanically enforceable" — is an insightful observation that the plan-approval gate depends on conversation state, not SQL. Also caught that 5c references 5d even when Medium skips 5d (process noise).
- MFA clarity rated 8/10. Good "why high / why not 10" structure but analysis is briefer.
- Loop start diagnosis: "internally structured yes, externally unambiguous no" — correct but less granular than Claude models.

### Codex — 40/50

- No regression, no gain — same as v0.10.15 baseline.
- 15 pre-code actions (fewer than other models but correct order).
- Identified the core 8 gates without counting size variants separately.
- Failure modes cover MFA sequencing, 3a dual gate, and 5c materialization — reasonable but without deep root cause analysis.
- Still dominated by MFA branching complexity and Step 5c mechanical load.

## Regression Analysis

The -0.75 average drop is concentrated in Sonnet (-2) and GPT-5.4 (-1). Root cause: the investigation continuation logic in Step C added a new branch to an already-dense decision tree. Models that previously scored the Step C complexity as manageable now identify it as more challenging. Opus and Codex were unaffected — Opus because it already tracked every branch perfectly, Codex because it was already at its scoring floor for Step C analysis.

**Verdict:** The regression is minor and expected — the fix adds necessary complexity to solve a real UX bug. The alternative (keeping the broken behavior where every investigation follow-up triggers full MFA re-entry) is worse than a 0.75-point benchmark drop. No action needed.
