# Benchmark — v0.10.15 Arm C

**Date:** 2026-04-14
**Version:** 0.10.15
**Arm:** C (candidate spec, terse overlay)
**Protocol:** Standard simulation prompt + concise overlay
**Baseline comparison:** v0.10.14 Arm C (2026-04-14)

## Changes under test (v0.10.15)

- **Continuation `task_id` hardening**: MFA Step C now explicitly discards the prior `task_id` on both new-task exits, Step D states that every new-task path generates a fresh slug, and the Verification Ledger clarifies that only true continuations reuse MFA Step C's value.
- **Evidence Bundle readiness tightening**: Step 5e now uses `COUNT(DISTINCT check_name)` and excludes `tier3-infeasible`, so duplicate rows and infeasibility bookkeeping no longer satisfy readiness.

## Notable Findings

### Opus — 50/50 (perfect, Arm C)
- Reads the new 5e gate correctly: distinct checks only, with procedural bookkeeping excluded.
- Main remaining complaint is no longer 5e; it shifted to **silent-step auditability** and the general complexity of continuation routing.
- Still considers Step 5c the highest mechanical compliance burden in the system.

### Sonnet — 47/50 (Arm C)
- Improvement comes from the continuation-hardening language landing clearly under the terse overlay.
- Still flags the **start signal after pre-signal `ask_user`** path as the most likely source of "loop not starting" reports.
- Continues to identify Step 5c render order as the major downstream risk once startup succeeds.

### GPT-5.4 — 45/50 (Arm C)
- Correctly quoted the final 5e query, including the new `tier3-infeasible` exclusion.
- Still sees "internal loop start" vs. "user-visible loop start" as a real ambiguity.
- Small regression vs. v0.10.14 Arm C because the benchmark attention shifted from the old 5e complaint to the remaining startup/start-signal ambiguity.

### Codex — 42/50 (Arm C)
- Stable versus v0.10.14.
- Same three themes as before: MFA sequencing, continuation complexity, and Step 5c machinery.
- No recurrence of the old raw-row 5e finding.
