**Date**: 2026-04-11
**Version**: 0.10.7 (continuation reliability + MFA forward-reference elimination)
**Panel**: claude-opus-4.6, claude-sonnet-4.6, gpt-5.4, gpt-5.3-codex

**Files evaluated**: `agents/odin.agent.md` (810 lines) + `skills/odin-review-prompts/SKILL.md` (232 lines)

## Changes from v0.10.6

1. **Objective completed-task detection in MFA step 3**: Before fuzzy scope checks, queries for terminal rows (`task-complete`, `investigation-complete`). If prior task finished → new task.
2. **Terminal `task-complete` row in Step 8**: Commits now INSERT `task-complete`, giving MFA step 3 an objective completion signal.
3. **Continuation start signal**: Continuation paths emit `> 🔁 **Odin Loop** — {task_id} | Resuming at Step {N}…`.
4. **Inline CREATE TABLE in MFA step 2**: Schema inlined directly instead of forward-referencing Verification Ledger section.
5. **Parallel-call clarity for first tool batch**: "first tool-call batch must contain both" replaces ambiguous "after setting" phrasing.

## Cross-Model Consensus

### v0.10.6 findings confirmed FIXED (no longer flagged)

1. ✅ **Forward reference in MFA step 2** — zero models flag the CREATE TABLE reference (was Opus v0.10.6 finding). Inlining the schema worked.
2. ✅ **`report_intent` vs `SELECT 1` ordering ambiguity** — Codex now rates MFA 9/10 (up from 8). The "first tool-call batch must contain both" rewrite eliminated the sequencing question. No model flags ordering ambiguity.
3. ✅ **Continuation missing start signal** — no model flags the invisible loop-start on continuations (was Sonnet v0.10.6 finding). The continuation signal addresses this.
4. ✅ **Continuation misclassifying completed tasks** — Sonnet explicitly identifies the completion check query as helping. The terminal-row detection is working as intended.

### Remaining consensus findings

1. **First-batch constraint is unenforceable** (all 4 models, persistent finding) — "counter to LLM training" (Opus), "prompt-following vs trained-behavior conflict" (Sonnet). Every model notes that LLMs instinctively read files before running the runtime gate. No spec-level fix exists — this is a training-vs-instruction tension inherent to LLM agents. Severity: structural limitation, not a spec clarity issue.

2. **MFA block density increased** (Opus + Sonnet) — Inlining the CREATE TABLE fixed the forward-reference issue but made MFA longer (~58 lines). Opus: "position is good but length is bad." Sonnet: "no single done signal for MFA." Trade-off: forward-reference compliance risk (v0.10.6) vs density/length risk (v0.10.7). The forward-reference fix is likely net positive (models must see the SQL to execute it).

3. **Step 0 sub-step overload** (Opus) — Step 0 alone contains 10 sub-steps (loop-entry gate, instruction scan, boost, sizing, start signal, ambiguity gate, non-overridable list, pushback gate, startup routing, investigation shortcut). This creates "gate fatigue." Not new to v0.10.7 but newly articulated.

4. **Continuation scope checks remain subjective** (Sonnet + GPT-5.4) — Despite the objective completion check (which catches completed tasks), the "same files / same feature / same bug" scope evaluation is still judgment-based. LLMs "consistently over-match on continuations" (Sonnet). The completion check helps the most common case but doesn't eliminate all fuzzy judgment.

5. **Skill invocation timing fragile** (Opus) — By Step 5c, the skill invocation instruction may be thousands of tokens from attention. Models may construct review prompts from memory rather than loading the skill. Not easily fixable without extracting more skills (which hits the 3-skill ceiling).

6. **Reviewer INSERT can be skipped** (Sonnet, novel) — After a reviewer returns findings, the LLM processes feedback and proceeds to fix issues — forgetting to INSERT the verdict first. Later, the 5c gate fires (returns 0), and the LLM may insert a synthetic row to pass the gate rather than re-running the reviewer. This is the exact hallucinated-verification failure the ledger is designed to prevent.

### Non-actionable findings (structural limitations)

- **First-batch constraint** is inherently unenforceable via instructions alone (all 4 models agree). This would require runtime-level enforcement.
- **GPT models plateau at 43-44** — consistent with v0.10.6. Shallower structural analysis in Failure Modes and Loop Start dimensions. Model capability, not spec clarity.
- **Skill invocation timing** is a fundamental context-window issue, not a spec-writing issue.

### Delta analysis (v0.10.7 vs v0.10.6)

- **Opus**: 49→49 (0). Same score. Different findings — old issues fixed, new ones identified (MFA length, gate fatigue). Net wash.
- **Sonnet**: 49→47 (-2). MFA density concerns (6/10 self-rating, down from 9/10 in v0.10.6). The inlined CREATE TABLE and added completion check made MFA longer. Sonnet's novel "no done signal for MFA" finding is legitimate.
- **GPT-5.4**: 44→44 (0). Stable. Same structural analysis depth.
- **Codex**: 42→43 (+1). MFA clarity improved (9/10, up from 8/10). The parallel-call rewrite specifically helped this model.
- **Net**: -0.25 from v0.10.6. Essentially flat — the v0.10.7 changes fixed 4 specific issues but introduced MFA density concerns that offset the gains for Sonnet.

### Interpretation

v0.10.7 successfully fixed its targeted issues (forward reference, ordering ambiguity, completion detection, continuation signal). However, the MFA block grew from ~30 lines to ~58 lines through the inline CREATE TABLE and new completion-check logic. This density increase hurt Sonnet's MFA clarity rating specifically. The trade-off is correct — forward-reference compliance is more important than block length — but further MFA compression should be explored in a future version.

The panel average of 45.75 matches v0.10.4/v0.10.5 and is within the measurement noise band (±0.5) of v0.10.6's 46.0. The hardening cycle from v0.10.2 (43.5) to v0.10.7 (45.75) represents a +2.25 improvement across 6 versions, with the gains primarily driven by structural reorganization (v0.10.4) and MFA unification (v0.10.5).
