## Change Under Test

Pure structural reorder of `agents/odin.agent.md` — no content additions or removals:

1. **Moved "On Every Message — Route, Execute, Guard" to the first section** after YAML frontmatter (was below personality)
2. **Moved "Intent Router" + write-time backstop** immediately after, before personality section
3. **Personality section now comes after routing and backstop** — models hit procedural gates before building persona
4. **Added descriptive section headers** ("How Odin decides what to do when", "Who you are, how you work, and the rules you follow")

**Motivation:** GPT-5.3-Codex failed catastrophically in a live session — read personality paragraphs, understood the task, and started executing destructive shell commands (`git reset --hard`, `rsync`, `composer create-project`) without ever entering Step 0. The hypothesis: moving procedural gates above personality forces models to encounter the routing algorithm before they start acting on the persona.

## Notable Observations

### The reorder is visible to GPT models and changes their MFA assessment

Codex explicitly cited "top-level route/execute/guard block" as a strength when rating MFA Clarity at 8/10 — this is a direct response to the structural reorder. In v0.12.3, Codex rated MFA Clarity at 7. GPT-5.4 also showed improved sequence detail, suggesting the earlier positioning of routing logic improved its ability to reconstruct the full pre-code sequence.

Claude models were already scoring at or near ceiling on these dimensions, so the reorder didn't move their totals — but both Opus and Sonnet rated the file's MFA clarity lower (6/10) than before (9 and 8 respectively). This isn't a regression in the benchmark score (the rubric scores analysis quality, not the rating given), but it reveals that the reorder surfaced a new concern: the "On Every Message" block is a compact code block, not prose, and models with stronger structural analysis now flag this as a processability risk.

### Opus identified a new concern: code-block format vs prose instructions

Opus's MFA analysis noted that the opening "Route → Execute → Guard" block uses arrows and bullets in a code fence, which LLMs may parse as a comment/summary rather than an executable procedure. This is a new finding that wasn't possible before because the block wasn't at the top. It suggests a follow-up: convert the compact code block to a numbered prose checklist.

### The write-time backstop gap remains unfixed

All 4 models correctly identified the write-time backstop. None flagged the specific gap (backstop only names `edit`/`create`, not destructive shell commands like `git reset`, `rsync`, `rm -rf`) because the benchmark prompt doesn't test against that scenario. The backstop text expansion is a separate fix from this structural reorder.

### Stale-task trap in Step 0a is a convergent finding

3 of 4 models (Opus, Sonnet, GPT-5.4) independently flagged the Step 0a continuation check as a source of loop-start failure — specifically, stale open tasks from abandoned conversations causing unwanted disambiguation prompts. This was also flagged in v0.12.3 but persists.

## Interim Verdict

**Net result: modest improvement (+0.50 avg), no regressions, GPT models benefit most.**

The structural reorder achieves its primary goal: GPT models now encounter routing and guard logic before personality, and this is visible in their improved MFA Clarity scores. The hypothesis that "personality before procedure" contributed to the GPT-5.3-Codex catastrophic failure is supported — Codex now explicitly identifies the top-level routing block as a strength.

However, the improvement is incremental, not transformative. The benchmark doesn't test the actual failure scenario (destructive shell commands bypassing the loop), so the real validation will be a live Codex session on a similar task. The benchmark confirms the reorder doesn't hurt and modestly helps.

**Recommended follow-ups:**
1. Fix the "Whow" → "Who" typo in the personality section header
2. Expand the write-time backstop to name destructive shell commands (`git checkout`, `git reset`, `rm -rf`, `rsync`, etc.)
3. Consider converting the opening code-block to a numbered prose checklist (Opus finding)
4. Address Step 0a stale-task trap (3/4 consensus finding)
