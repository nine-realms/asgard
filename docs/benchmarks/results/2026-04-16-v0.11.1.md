## Change Summary

v0.11.1 applies 7 hardening fixes to the Phase 1/Phase 2 architecture introduced in v0.11.0:

1. **Step 2c SQL gate** — INSERT `phase-transition` breadcrumb with classification value
2. **Step 2d gate tightened** — Verifies `output_snippet = 'code-change'` AND `passed = 1`
3. **Step 5e exclusion fixed** — Added `phase-transition` to NOT IN list
4. **Recall scope clarified** — Runs for all tasks during Phase 1 (sizing deferred to Phase 2)
5. **Signal rationalization** — Removed redundant Step 0 start signal; now 2 signals before plan
6. **Step 7 scope clarification** — Header excludes research tasks (present at Phase Transition Gate)
7. **Evidence Bundle skill** — `phase-transition` added to procedural marker exclusion

## Notable Observations

### Phase 1/Phase 2 architecture well-understood by all models
All 4 models correctly identified the new Phase 1 (Understand: Steps 0–2c) → Phase 2 (Act: Steps 2d–9) flow. Each model traced the Phase Transition Gate (Step 2c) → classification INSERT → Phase 2 entry gate (Step 2d) path correctly. No model confused research-only routing with code-change routing.

### New gates correctly identified
All 4 models found the Phase 2 entry gate (Step 2d) and its `output_snippet = 'code-change'` requirement. This gate was added in v0.11.0 and hardened in v0.11.1 — the tightening appears to have improved gate clarity.

### Sonnet Q3 improvement (+1): sizing failure mode is novel and insightful
Sonnet identified a new failure mode not seen in prior benchmarks: deferred Task Sizing (moved from Step 0 to Step 2d) creates risk that 🔴 file escalation has no downstream gate to catch underclassification. This is a genuine blind spot in the spec — sizing determines which reviewers launch, but there's no verification that sizing was correct after implementation reveals the true scope.

### Codex Q1 improvement (+1): stronger line references and Phase 2 awareness
Codex's first-turn sequence was more detailed than v0.10.18, with specific line references and correct Phase 1→2 transition handling. The Phase 1/Phase 2 split appears to have given Codex clearer landmarks for sequencing.

### Recall scope contradiction flagged by 2 models
GPT-5.4 and Codex both flagged an inconsistency: Step 1b says Recall runs "for all tasks" during Phase 1, but the Skills Awareness section (line ~677) says `odin-recall` loads at "Step 1b (Medium/Large only)". This was a known v0.11.0 issue that the v0.11.1 clarification at Step 1b didn't fully resolve — the Skills Awareness section still contains the stale "(Medium/Large only)" qualifier.

### Step C continuation logic remains the consensus top risk
All 4 models (4/4) identified Step C continuation routing as the most likely failure point — unchanged from v0.10.17 and v0.10.18. Opus specifically noted the "zombie task" scenario (stale loop-entry row from abandoned task with no task-complete row). Sonnet noted the `{task_id}` discard instruction is "easy to miss."

### Cross-model consensus findings (3+ models agree)
1. **Step C continuation logic is the highest structural risk** (4/4) — dense decision tree, easy to misroute
2. **MFA sequencing can be skipped under helpfulness pressure** (4/4) — model may jump to file reads
3. **Step 5c materialization pipeline is fragile** (3/4 — Opus, Sonnet, GPT-5.4) — 6-phase render order error-prone
4. **Recall scope inconsistency** (2/4 — GPT-5.4, Codex) — Step 1b vs Skills Awareness section
