## Changes under test (v0.10.13)

- **Step 0 gate reordering**: Moved ambiguity gate, non-overridable behaviors, and pushback gate to execute *before* task sizing and start signal (was after). Start signal wording updated to "After all gates resolve — ambiguity, pushback, and sizing (including any Investigation confirmation)."

- **Escalation hard stop**: Step 3 size-escalation path converted from advisory text to `🚫 Stop` gate with explicit prohibition on proceeding to Frigg and a `context-gathered` re-insert instruction for auditability.

## Notable findings

### Cross-model consensus (2+ models)

**MFA as sibling `##` heading** (Opus, Sonnet): Both models independently identified that MFA (`## ⚠️ MANDATORY FIRST ACTIONS`) is a structural sibling to `## The Odin Loop` rather than a subsection of it. LLMs scanning for "where to start the loop" land on `## The Odin Loop` → Step 0 and may treat MFA as preamble. Step 0's recovery instruction ("return to MFA step A") is a soft advisory, not a hard gate.

**Step 5c prompt materialization fragility** (all 4 models): All four identified the 6-phase render order in `odin-review-prompts` as a high-risk compliance point — specifically: unresolved `{IF_SPEC_FILES_IN_DIFF}` conditionals left verbatim, `{panel_list}` token not substituted on Small tasks, and LLMs launching reviewers with malformed prompts rather than halting.

**Step C continuation misclassification** (Sonnet, Codex): If a prior session left a `loop-entry` row in the same session DB, the new task may be misclassified as a continuation, skipping D–E and leaving the new `task_id` without a loop-entry row. All downstream gate queries then silently return 0.

### Novel findings (single model, high-signal)

**skill() vs system-prompt contradiction** (Opus Q3, high severity): The agent file says "Call `skill('odin-review-prompts')` directly — do not gate on `<available_skills>`." The runtime system prompt says "Only use skills listed in `<available_skills>`." These directives directly contradict. An LLM that weights system-prompt instructions above agent-file instructions will refuse the skill call, triggering HALT and blocking all adversarial review. This is a genuine behavioral spec defect.

**Plan approval gate has no SQL tripwire** (Sonnet Q3): The 3a Frigg-recorded gate has a SQL verify. The second 3a gate (conversation must contain `ask_user` after Frigg INSERT) has no verification mechanism — it's purely procedural. A model can reason "the user said proceed" from context and skip the `ask_user` call, especially on Small tasks. No SQL check catches the failure.

**54-line identity block as buffer dilution** (Opus Q5): After MFA-E's "immediately begin Step 0" instruction, 12 lines of identity/philosophy prose separate it from `## The Odin Loop`. This buffer reduces transition signal salience. A model may treat the identity block as the "response" to MFA completion and generate prose output rather than continuing to tool calls.

## Interpretation

The v0.10.13 fixes produced a modest avg +1.5 gain, primarily driven by Codex (+5) — the explicit gate ordering and hard stop language reduced structural inference requirements for task-oriented models. Sonnet gained +2. Opus and GPT-5.4 were within noise.

The highest-leverage unresolved issues surfaced by this run:
1. MFA structural placement (sibling `##` vs child `###`) — affects all models
2. `skill()` vs system-prompt contradiction — blocks Step 5c on system-prompt-compliant models
3. Plan approval has no SQL tripwire — silent compliance failure on the most user-visible gate
