## Notable Findings

### 1. MFA bypass risk — rated "High likelihood" (Opus)

The file has two prominent section headers: `## ⚠️ MANDATORY FIRST ACTIONS` and `## The Odin Loop`. LLMs pattern-match `## The Odin Loop` as the entry point and treat MFA as setup metadata. Step 0's precondition check is inside Step 0 — meaning the LLM already decided to start there before the check fires.

**Recommendation (Opus):** Add a `🚫 GATE` at the top of `## The Odin Loop`, before Step 0, requiring a verified `loop-entry` row:
```sql
SELECT COUNT(*) FROM odin_checks WHERE task_id = '{task_id}' AND check_name = 'loop-entry';
-- If 0, return to MANDATORY FIRST ACTIONS
```
This leverages the existing gate pattern the LLM already follows reliably, rather than relying on prose precondition text.

### 2. Frigg double-gate asymmetry (Sonnet)

The normal Frigg path (INSERT → ask_user) and the timeout path (INSERT timeout-row → INSERT user-decision-row → ask_user) have different INSERT counts before the user-approval gate. The timeout path's "do not prompt a second time" note (for the Frigg-surfaced-tradeoff case) risks being over-generalized — an LLM may apply it to the normal path and skip ask_user entirely after inserting the Frigg row.

### 3. Step 5c semantic error escapes phase 6 verify (Opus)

The `review_context=panel → standalone` rewrite for Small tasks (phase 2) is a semantic transformation. If phases 1–5 collapse into one substitution pass, the model may substitute `{panel_list}` with an empty string rather than leave it unresolved — which passes the phase 6 unresolved-placeholder check but sends Mimir a malformed prompt (`panel_reviewers=`).

### 4. SKILL.md naming mismatch in file classification (GPT-5.4)

The `odin-review-prompts` skill may describe spec files as `*.skill.md` for detection purposes, while this repo's actual files are named `SKILL.md`. A mismatch here would cause skill file changes in a diff to be misclassified (code prompt instead of spec prompt), applying the wrong review criteria.

### 5. MFA "in this turn" ambiguity (Sonnet)

The sentence "Steps B–E follow immediately in this turn, each as its own tool-call batch" packs batch composition, exclusions, sequencing, and turn vs batch distinction into one clause. "In this turn" may be interpreted as "all in one LLM response," which breaks the C→D conditional dependency (you cannot run D until you've read C's result).

### 6. 52-line attention gap between MFA entry and Loop start (Opus)

From "Every message enters here" to "immediately begin Step 0" spans ~38 lines of conditional SQL and branching logic. The strong `## ⚠️` signal at line 8 may be "resolved" by the LLM's attention before the procedural chain completes, leading to conceptual acknowledgment without step-by-step execution.

## Q4 Pattern — "Trailing catch vs precondition"

All four models rated Q4 at 7–8/10. The shared critique: the MFA enforcement hook ("if you are about to call `edit`... stop and return to MFA step A") is positioned as a trailing reminder after all steps, not as a precondition before them. A precondition at the literal first sentence of the loop would be more robust.
