## Analysis

### Score-neutral — within noise band

The −0.75 average drop (45.25→44.5) is within the ±1.5 noise band established by prior same-day reruns. The 6-line spec change (+6 from Frigg fix) is too small to produce a structural shift.

**Interpretation: no regression.** The v0.10.10 changes are confirmed neutral.

### All models correctly read the new `AND passed = 1` gate

Every model that enumerated the Step 3a Frigg gate query included `AND passed = 1` — confirming they read the v0.10.10 file, not the old version. This is the primary change being tested.

The Frigg gate fix was also independently validated: **Mimir** (during Step 5c review of this very change) flagged that a `passed=0` cancellation row would satisfy the old gate — the exact same structural bug that motivated the fix. Mimir's finding and the benchmark models' reading of the fix are consistent.

### Codex drop (42→39) is the sole driver of average decline

Codex scored noticeably lower on Failure Modes (8→7), MFA Clarity (8→7), and Loop Diagnosis (8→7). This is likely instance-level variance rather than a structural regression — Codex's response was complete and correct on the factual questions (sequence, gates), but provided less analytical depth on the interpretive questions this run. The −3 from Codnet accounts for the entire apparent average decline.

### Sonnet improvement: 46→47 (+1)

Sonnet recovered from its v0.10.9 dip. Two new findings not seen in other models:

1. **Break Point 3 (ambiguity gate before start signal)**: identified as the *most likely common cause* of "loop not always starting." The start signal fires "immediately after sizing" but the ambiguity gate fires "after boosting" — and since boosting precedes sizing, an LLM resolving ambiguity before sizing will emit `ask_user` before the start signal appears. The user sees a clarifying question with no loop signal and perceives the loop as not having started. Fix: move the start signal before boost as a commitment token.

2. **Break Point 4 (MFA re-run → ledger corruption)**: if the Step 0 gate triggers an MFA re-run, the continuation check in Step C may find the old `loop-entry` row and classify the request as a continuation on the wrong `task_id`. Old verification rows may then spuriously satisfy gate queries. This is a subtle multi-step failure not identified by other models.

### Shared cross-model finding: "Nothing else" stall

Three of four models (Opus, Sonnet, GPT-5.4) independently identified the `"First batch = X + Y. Nothing else"` phrasing as ambiguous. An LLM may read "Nothing else" as "do these two and yield" rather than "do these two, then continue through B–E in the same turn." Opus proposed a concrete structural fix (rewrite as an explicit numbered multi-step with a "→ Proceed to Step 0. Do not stop." directive). Sonnet proposed moving the start signal before boost as an alternative commitment mechanism.

This is a **high-confidence finding** (3/4 model convergence). The current spec text is structurally ambiguous on this point. Addressing it is the highest-priority improvement for v0.10.11.

### Opus Loop Start: 9→10 (+1)

Opus improved its Loop Start score by identifying three named break points with concrete fixes, including an explicit structural rewrite of the MFA section that would eliminate the "Nothing else" stall. Previously Opus diagnosed the causes; this run it also proposed the remedy.

### MFA Clarity remains the weakest dimension (7.75 avg)

Consistent across versions. The Step C continuation logic complexity and the "Nothing else" stall are both cited by multiple models as the primary sources of ambiguity. The A–E rename (v0.10.10 change) addressed the namespace collision but did not address the prose complexity in Step C or the "Nothing else" phrasing.
