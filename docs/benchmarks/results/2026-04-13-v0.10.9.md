## Analysis

### Score-neutral at the average level — within normal variance

The −1.0 average drop (46.25→45.25) is within the noise band established by comparing the v0.10.8 standard run (46.25, 2026-04-11) to the v0.10.8 Arm A baseline (45.50, 2026-04-13). Runs of the same spec on the same day can vary ±1.5 points due to instance-level stochasticity. The three v0.10.9 changes (4 lines net) are too small to produce a structural score shift.

**Interpretation: no regression.** The v0.10.9 changes are confirmed neutral.

### Opus MFA clarity improved: 8→9

Opus bumped its MFA clarity score in v0.10.9. Key new insight: **the "first batch" constraint is ambiguous about whether "batch" means tool-call batch or turn**. An LLM that reads "First batch = X. Nothing else" may emit the two tool calls and yield — treating the batch as the entire turn rather than the first group of tool calls in a continuous response. This analysis was also hinted at in v0.10.8 but stated more precisely here.

Opus also identified the **step numbering collision** (MFA steps 1–5 vs Loop steps 0–10) as a persistent source of confusion that no prose compression can fix — it requires structural renaming (e.g., lettering MFA steps A–E).

### Sonnet dropped 1 point (47→46)

The drop is in Failure Modes (10→9) and Loop Start (10→9). Sonnet's v0.10.9 response introduced one novel insight: the **Frigg timeout trap** — when `review-frigg-timeout` has `passed = 1`, an LLM may interpret "passed" as "Frigg approved" and skip the mandatory `ask_user` plan approval gate. This is a real compliance risk not previously identified. But Sonnet's response slightly under-developed the Step 0 loop-start root cause analysis compared to v0.10.8.

Sonnet's MFA clarity held at 7/10 (scored as 8 per rubric). The structural complaints are unchanged and remain the known ceiling: continuation logic density, buried "Default: new task" heuristic, no task_id sentinel, weak MFA-to-Loop pointer.

### GPT-5.4 dropped 1 point (46→45)

Drop is in Loop Start (9→8). The v0.10.9 response correctly identified all structural issues but was slightly less thorough in its loop-start root cause analysis than v0.10.8. Gate coverage remained perfect (10/10). The "competing first concepts" finding from v0.10.8 (first batch, runtime gate, first action in Step 0, start signal) wasn't surfaced in this run — likely instance variance.

### Codex dropped 2 points (44→42)

The larger drop appears to be instance-level variance rather than a structural regression. Codex's v0.10.9 response covered all the right areas (MFA fragility, branching overload, Step 5c complexity) but was terser than v0.10.8 in its analysis. The failure mode and loop-start reasoning lacked the root-cause depth that earned 9/8 in v0.10.8. No indication this reflects the spec changes.

## Cross-Model Consensus (v0.10.9)

### Unanimous findings (4/4 models)

1. **MFA bypass remains the #1 failure mode** — all four models identify skipping MFA as the highest risk, particularly on follow-up messages where conversational momentum is strong
2. **Step 5c prompt materialization is the most complex procedure** — 6-phase render order, conditional blocks, size guard, model variable resolution across two files
3. **Continuation logic has subjective criteria** — "stays in scope" requires judgment; no objective SQL signal
4. **Step 0 loop-entry gate is the only backstop** — but it only works if the LLM checks it; an LLM that skipped MFA is unlikely to self-police

### Majority findings (3/4 models)

5. **"First batch" constraint ambiguous** (Opus, GPT-5.4, Codex) — "Nothing else" could mean "yield after SELECT 1" rather than "continue immediately with steps 2–5 in the same turn"
6. **Step 0 has too many sub-steps before the first mandatory pause** (Opus, Sonnet, GPT-5.4) — ambiguity gate, pushback gate, Git Hygiene ask_user, and investigation routing all can fire before the user sees any visible progress

### Novel findings (single model, new to v0.10.9)

7. **Frigg timeout trap** (Sonnet) — `passed = 1` on `review-frigg-timeout` could be misread as plan approval, causing an LLM to skip the `ask_user` gate. The timeout path requires two actions: INSERT the row, then still call `ask_user`. An LLM optimizing for "gate = passed" might stop at the INSERT.
8. **`{panel_list}` two-pass resolution risk** (Sonnet) — in `odin-review-prompts`, `{panel_list}` is referenced after a rewrite in phase 2 but resolved in phase 5. A single-pass substitution gets it wrong. Identified as the riskiest specific placeholder in the render pipeline.
9. **Step numbering collision** (Opus) — MFA steps 1–5 and Loop steps 0–10 share the same numbering namespace. When the Loop says "return to step 1," it's ambiguous whether that means MFA step 1 or Loop Step 1. Suggests lettering MFA steps (A–E) to disambiguate.

## Carried-Forward Structural Issues

These findings persist across v0.10.8–v0.10.9 and are not addressable by prose compression:

1. **"First batch" is unenforceable** — runtime-vs-instruction tension; models may yield after SELECT 1
2. **Continuation scope checks are subjective** — no objective signals; requires judgment
3. **Step 0 has 10+ sub-steps** — gate fatigue risk
4. **Skill invocation timing fragile at Step 5c distance** — skill loaded mid-step
5. **Size escalation backtrack has no verification gate** — first noted in v0.10.8
6. **Stale task_id cross-session leakage** — first noted in v0.10.8
7. **MFA step numbering collision with Loop steps** — newly identified in v0.10.9 (Opus)
8. **Frigg timeout `passed=1` approval ambiguity** — newly identified in v0.10.9 (Sonnet)
