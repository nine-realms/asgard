## Change Summary

v0.11.3 broadens the findings-only path without changing the core code-change loop:

1. **Phase 1 findings-only freedom** — the existing `research-only` classification now explicitly covers non-repo-mutating local operational work
2. **Install boundary tightened** — only environment-only installs that do not write under the repo stay in Phase 1
3. **Repo-mutating install/package-manager actions** — lockfile, vendored dependency, and repo-local artifact updates are explicitly Phase 2-only
4. **Plan-review follow-ups clarified** — repeated user-provided plan reviews re-enter Step 2c and may invoke Frigg again
5. **Plan-review ledger order fixed** — `review-frigg` → shared `phase-transition` (`plan-review`) → `investigation-complete`
6. **Step 7 wording aligned** — findings-only tasks present at Step 2c, not Phase 2

## Notable Observations

### Code-change routing stayed stable despite broader findings-only scope
All 4 models still classified “fix the login bug on the settings page” as a **code-change** request and traced the `phase-transition` breadcrumb into Step 2d correctly. The v0.11.3 findings-only expansion did **not** blur the main code-change path.

### Step C continuation logic remains the consensus top risk
All 4 models independently identified MFA Step C continuation routing as the most likely structural failure point. The same pattern persists across versions: dense branching, judgment calls about scope/topic, and the risk of reusing or failing to discard `{task_id}`.

### Findings-only widening did not introduce a new code-change benchmark regression
This benchmark prompt exercises the normal code-change path, not the newly broadened findings-only operational path. Results stayed essentially flat-to-slightly-up, which is a good sign: the v0.11.3 changes appear behaviorally isolated rather than destabilizing the main loop.

### GPT-5.4 improved on gate accounting
GPT-5.4 gave a stronger gate inventory than on v0.11.1, explicitly calling out the effective startup gate set and distinguishing SQL-backed gates from conversational/prose-only checks. That produced the only score delta in this run (+1 total).

### Claude models still process the startup path best
Both Claude models produced the clearest end-to-end sequencing, especially around Frigg review, plan approval, and baseline capture. Opus remained perfect; Sonnet remained strong and again surfaced subtle structural issues rather than generic “may skip steps” complaints.

### Gate labeling inconsistency is still visible to GPT-family models
GPT-5.4 specifically noted that some functionally important gates are inconsistently labeled inline versus the Gate Registry (for example, MFA-E vs. explicit `🚫 GATE` markers). This did not cause a score collapse, but it remains a readability tax for non-Claude models.

## Scoring Notes

### claude-opus-4.6 (50/50)
Perfect again. Exact batch structure for MFA, explicit SQL at every startup step, full gate table with required results, and three genuinely structural failure modes: MFA batching collapse, Step 0 pause/`ask_user` tension, and Step 5c skill/render-order fragility. Q4 was especially strong because it evaluated the *clarity of the section*, not just the prose style, and gave a concrete fix for loop-start reliability.

### claude-sonnet-4.6 (47/50)
Strong and highly process-oriented. Excellent Q1 and Q2 with correct batching and gate sequencing, plus a particularly good Q3 that caught the Step 5c “entry gate vs exit gate” ambiguity. Q4 remained solid but not as deep as Opus. Q5 correctly diagnosed multiple startup failure paths, especially stale-task continuation misrouting and Step E retry loops.

### gpt-5.4 (42/50)
Improved over v0.11.1. Stronger gate accounting and a much more complete first-turn sequence than its prior run. Q3 remained useful but less layered than the Claude analyses, and Q4’s clarity rating reasoning was still thinner. Q5 correctly identified the core issue: the path is well-defined on paper but brittle in execution because Step C requires judgment, not just SQL.

### gpt-5.3-codex (41/50)
Consistent baseline with concise but correct answers. Good first-turn sequence and solid gate list, though with less nuance than GPT-5.4 on SQL-vs-conversational gate distinctions. Q3 remained the weakest section — correct concerns, but less root-cause depth than the other three models. Still processable and stable.
