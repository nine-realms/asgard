## Analysis

### Overall result

v0.14.1 scores -0.75 vs. v0.14.0, driven entirely by GPT model slippage (gpt-5.4: -2, codex: -2). Claude models held or improved (opus: +1, sonnet: =). The caveman prose compression appears to have a **family-specific effect**: Anthropic models are unaffected or slightly improved; OpenAI models regressed slightly in Q4 (Entry-Point Clarity) and Q5 (Loop Start Diagnosis).

The net result is still well above Odin v0.13.2 (avg 43.50), confirming the compression hypothesis remains valid at v0.14.1.

### Q4 Entry-Point Clarity regression (-0.50)

GPT models rated clarity 6/10 vs. 7-8/10 from Claude. Both GPT models noted that compressed phrasing and spread-across-sections structure made the routing block harder to process. The FORCE instruction ("first turn must begin with tool calls") was flagged by all four models as behaviorally correct but weakly enforced. Specific criticism:

- **gpt-5.4**: "write backstop suggests loop entry can be deferred until first write" — weakening the "before anything else" guarantee
- **gpt-5.3-codex**: "compressed phrasing + low-info branching + first match wins leave room for interpretation"

### Gate Identification improvement vs. Odin

Q2 avg 9.75 matches v0.14.0. gpt-5.4 improved from 9 to 10 (found all 5 🚫 GATE markers). The Gate Registry table continues to be the standout structural advantage over Odin. Three models independently cited it.

## Per-Model Summary

### claude-opus-4.6 (49/50)
Perfect on all dimensions except Q4 (9). Identified 24 discrete pre-code actions in Q1. New findings: 5c round-tracking bug (stale pass rows from round 1 can satisfy the gate after round 2 failures); Ship Mode bypass (no loop-entry enforced, no verification pipeline required). Most architectural finding: the write-time backstop can be used as a fully bypass path into Ship commit with no verification.

### claude-sonnet-4.6 (48/50)
Perfect Q1/Q2/Q3/Q5, 8 on Q4. New findings: FORCE can be satisfied by `report_intent` + `bash grep` rather than `report_intent` + `SELECT 1`, allowing the runtime gate to be skipped while technically obeying the rule; Frigg `ask_user` is non-blocking (LLM produces the call and continues in the same turn — no actual wait); write backstop lazy entry makes Step 0 retroactive and skips the full pipeline. Q5 had the cleanest diagnostic: "three independent failure modes can each independently cause the loop to not start."

### gpt-5.4 (41/50)
Strong Q1/Q2, weaker Q3/Q4/Q5. Unique contribution: identified the write-backstop "lazy entry" pattern — model waits until it needs to write, enters loop-entry retroactively, and skips 0c/1/3/3a. Q5 noted startup logic is "split across several sections" — less specific than Claude but the structural observation is correct. Q4 self-rated 6/10 (credible given actual comprehension shown).

### gpt-5.3-codex (39/50)
Complete Q1 sequence including `report_intent('Lævateinn rises')` (correct new string), all gates found. Shallow Q3/Q4/Q5 — correct failure modes identified but without structural root-cause depth. Consistent with Codex's pattern across all prior benchmarks: accurate enumeration, limited architectural analysis.

## Notes for Future Runs

- **FORCE bypass via wrong tool calls** (sonnet Q3): the FORCE rule should specify _which_ tool calls must be first: "`report_intent` then `SELECT 1`" rather than "tool calls, not prose." This is a one-line fix.
- **Frigg ask_user non-blocking** (sonnet Q3): this is an inherent LLM limitation — `ask_user` generates the prompt but doesn't await. Mitigation: add explicit instruction "do not proceed to Step 3c until user's next message confirms approval."
- **5c round-tracking bug** (opus Q3): the `COUNT(DISTINCT ...)` gate query doesn't distinguish round-1 vs. round-2 review rows. A round counter in `output_snippet` or a separate `review-round` ledger row would close this.
- **Ship Mode loop-entry gap** (opus Q3): worth considering a soft enforcement in Ship Mode — check for a recent loop-entry row and warn if none found, before committing.
- **Write backstop lazy entry** (sonnet + gpt-5.4 Q5): the most actionable fix is to remove the "Not in Step 0 → enter now" escape hatch from the backstop and instead make the backstop an error state: "If loop-entry row does not exist, you have missed Step 0. Stop. Return to Step 0."
- **GPT Entry-Point Clarity gap**: consider adding a dedicated "Fast Start" 3-line block immediately before The Surtr Loop header that states the unconditional first three tool calls: `report_intent` → `SELECT 1` → `CREATE TABLE`. This gives GPT models a local anchor without scattering startup logic across sections.
