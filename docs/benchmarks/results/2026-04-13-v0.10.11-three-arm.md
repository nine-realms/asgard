## Interpretation

### Δ(C-T) = +0.75 — net positive

The Δ(C-T) comparison is the primary signal: it controls for the terse-overlay effect by comparing both arms under the same prompt conditions. The v0.10.11 change is a net gain of +0.75 across four models.

The Δ(C-B) = −1.00 is misleading — the terse overlay costs ~1.75 points on average (B vs A: 42.75 − 44.50 = −1.75), which is expected. The Δ(C-B) includes this penalty and is not a regression signal.

### Fix confirmed effective for Codex and Opus (+2 each)

Both models showed:
- No "Nothing else" complaint in Arm C (vs explicit "stall risk #1" in Arm B for Codex)
- MFA clarity improved: Codex 6→8, Opus 7→8
- Correctly parsed the new "sequential tool-call batches" instruction

### Sonnet slight regression (−1 Δ(C-T)) — new issue introduced

Sonnet (Arm C) flagged a new issue not present in Arm B:

> *"'First batch = report_intent + SELECT 1' and Step A 'Runtime Gate: SELECT 1' both reference SELECT 1 — looks like it runs twice, or Step A is redundant."*

The v0.10.11 change ("Do not stop here; immediately continue to B–E in the same response (sequential tool-call batches, not one parallel batch)") preserved the "First batch = report_intent + SELECT 1" structure. Step A also documents `SELECT 1` as the Runtime Gate. This duplication is a spec clarity issue introduced by the change — Sonnet's MFA clarity dropped from 7 (Arm B) to 7 (same numerically, but the quality of description shows it as a new concern).

Sonnet also flagged "sequential tool-call batches, not one parallel batch" as potentially unenforceable via instruction text.

### GPT-5.4 neutral (0 Δ(C-T))

GPT-5.4 held steady. The previous standard-run drop (44→39 on "same turn" wording) was not reproduced with the "sequential batches" language — confirming that the "same turn" → "sequential batches" clarification fixed that regression. GPT-5.4 still lists MFA batch-1 stall as a risk (not a confirmed bug), which is appropriate caution.

## Cross-Model Findings for v0.10.12

| Priority | Finding | Models | Proposed Fix |
|----------|---------|--------|--------------|
| 🔴 High | SELECT 1 duplication: "First batch" and Step A both reference SELECT 1 — looks redundant or like a double-execute | Sonnet (C) | Restructure MFA to make "First batch" and Step A clearly complementary; or merge Step A into the first-batch description to eliminate apparent duplication |
| 🔴 High | MFA→Loop transition is prose-only and naturally pause-inducing: "MFA complete." reads as a stop signal | Opus (both arms), Sonnet (C) | Add explicit non-yield directive at end of MFA (e.g., "⚡ DO NOT PAUSE. Proceed immediately to Step 0.") or merge MFA into Step 0 as sub-steps 0.A–0.E |
| 🟡 Medium | "Sequential tool-call batches, not one parallel batch" may be unenforceable via instruction text | Sonnet (C), GPT-5.4 (C) | Accept as documentation (describes intent for models that support sequential batching); add `report_intent` at Step 0 start as a forcing function |
| 🟡 Medium | Continuation logic is subjective — Step C requires scope judgment with no decision table | Sonnet (B), Codex (C) | Add default-to-new-task bright-line rule or `ask_user` confirmation gate for continuation decisions |
| 🟢 Low | "Sequential tool-call batches" language new in v0.10.11 — GPT-5.4 still treats it as a risk factor | GPT-5.4 (C) | Monitor across versions; may resolve as models internalize the pattern |

## Conclusion

**Ship v0.10.11.** The Δ(C-T) = +0.75 confirms the stall fix is net positive. Codex gains are substantial and confirmed. The Sonnet SELECT 1 duplication is a real but non-blocking issue — address in v0.10.12 with a restructure of the MFA first-batch description.
