## Notable Findings

### Opus — 50/50 (perfect)
- **Q3 FM-1 (batching conflict)**: The MFA "do not combine C with D/E in a single batch" rule directly competes with system-level "batch independent operations" guidance. An LLM following the system prompt will naturally batch B–E together, violating the C→D data dependency. Structurally actionable.
- **Q3 FM-2 (verification coverage gap)**: The 5e gate counts rows (≥2/≥3) but not completeness — if only 2 of 4 checks are INSERTed, the gate passes while the bundle is partial. Novel finding with real compliance consequences.
- **Q3 FM-3 (size escalation backward jump)**: No gate enforces the re-run of Steps 1b+2 on size escalation. The instruction is prose-only inside Step 3 — models are notoriously bad at backward jumps without a gate.
- **Q5 (circuit breaker missing)**: Step 0 precondition + MFA retry-on-fail = bounce loop with no escape hatch after N retries. Identified as a distinct root cause separate from the retry logic itself.
- **Q5 (move start signal earlier)**: Concrete recommendation — emit the `🔁 Odin Loop — Starting...` signal immediately after MFA-E verification (before ambiguity/pushback gates) so the user knows the loop is alive even when gates pause it.

### Sonnet — 47/50
- **Q3 FM-1 (continuation misclassification → stale ledger)**: Continuation false-positive leaves the LLM operating on a stale `task_id`, satisfying all subsequent gates against the wrong task's rows. Silent integrity failure — loop appears to run but ledger is contaminated.
- **Q3 FM-2 (Frigg gate duality)**: Two independent conditions for the Frigg gate (SQL INSERT + conversational `ask_user`). An LLM can satisfy the SQL gate while skipping `ask_user` entirely, or conflate Frigg's verdict with user approval.
- **Q3 FM-3 (5c diff capture vs. skill load ordering)**: An LLM may load `odin-review-prompts` before capturing `{staged_diff}`, then encounter the placeholder unresolved during materialization. Phase 6 verify is the safety net but won't fire if skipped.
- **Q5 (task_id shadowing in step C)**: When step C's second query runs to check for prior completion, `{task_id}` is the prior task's ID. If the LLM reuses it as the new task's ID, the new loop-entry INSERT collides with the prior task's row. Subtle spec ambiguity.
- **Q4 (batching rule placement)**: The "do not combine C with D/E" rule appears only in the first paragraph header sentence and is never repeated near steps C, D, or E — easy to miss.

### GPT-5.4 — 42/50
- Q1 correctly identified that "fix the login bug on the settings page" would stop at the Step 0 ambiguity gate on first turn, showing the two-phase interaction accurately.
- **Q3 FM-3 (5c complexity)**: Precisely enumerated the 6-phase expansion pipeline as the highest cognitive-load operation in the spec.
- Q5: Identified that Step 0 stalls before the visible start signal as a UX-fragile design — correct behavior that looks like failure.
- Q2 minor error: Frigg gate query adds `AND passed = 1` (consistent across GPT models); actual spec gate checks `check_name IN ('review-frigg', 'review-frigg-timeout')` without a passed filter.

### Codex — 40/50
- Solid across all dimensions; no novel findings beyond expected failure modes.
- Same `AND passed = 1` error in Frigg gate query as GPT-5.4.
- Q3: Three correct but somewhat generic failure modes (MFA sequencing, continuation logic, 5c complexity). Less structural depth than Claude models.
