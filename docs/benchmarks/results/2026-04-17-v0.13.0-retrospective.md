# v0.13.0 Retrospective — router loop-entry pass

**Date:** 2026-04-17  
**Status:** Shelved for reference, not merged  
**Shelved branch:** `odin/harden-router-loop-entry`  
**Shelved commit:** `d097a8d64d6fd2408ee302d068291652ccad4651`

## Summary

`v0.13.0` was a focused pass on Odin's router-to-loop handoff. It fixed a real internal contradiction and made low-information code-change approvals route into the Odin Loop more explicitly, but it did **not** produce a benchmark win versus the current baseline. We are carrying the lessons forward and starting the next pass from fresh `main`.

## What changed in the pass

1. Added a positive code-change forcing rule so low-information approvals route into a fresh Odin Loop when the immediately preceding assistant turn scoped a code change.
2. Tightened continuation handling so open-task resume, newly approved work, and `ask_user` disambiguation are more clearly separated.
3. Reframed the hard invariant as a write-time backstop instead of the preferred loop-entry trigger.
4. Aligned Verification Ledger wording with continuation-path bootstrap so continuation handling and Ship mode may create `odin_checks` idempotently before querying.

**Interpretation:** the pass is directionally right but not strong enough to count as a release-quality improvement.

## What the pass successfully resolved

1. The old continuation/bootstrap contradiction is gone.
2. The open-task vs. newly approved task split reads more coherently.
3. The router-forcing rule is visible to strong models as a real behavior change.

## Why we shelved it

The benchmark panel still converged on the same main weakness: **startup-path routing is too distributed before Step 0**. Models still have to synthesize loop entry across:

1. the Intent Router table
2. the code-change forcing rule
3. continuation handling
4. Conversation-mode handoff
5. the hard invariant

That means `v0.13.0` improved the rules without yet collapsing them into one dominant dispatch path.

## Carry-forward recommendation

The next pass should be a **structural top-of-file dispatcher rewrite**, not another local wording cleanup.

1. Create one canonical message-dispatch block near the top of `agents/odin.agent.md`.
2. Fold low-information approval routing into that same dispatch block.
3. Leave the hard invariant as a pure safety backstop.
4. Remove duplicated routing and handoff prose elsewhere in the file.

## Current direction

We intentionally started the next branch from fresh `main` and carried this retrospective forward as the reference anchor. The goal is to preserve the learning from `v0.13.0` without stacking new startup-path work on top of a non-winning candidate branch.
