## Analysis

### Score-neutral — within noise band

The −0.5 average (44.5→44.0) is within the ±1.5 noise band. This benchmark ran on the intermediate v0.10.11 wording ("same turn") before the "sequential batches" clarification was applied. The model distribution shifted but the average held.

### "Nothing else" stall fix — confirmed effective for Codex (+3) and Opus (+1)

The primary goal of v0.10.11 was to eliminate the "stall" where models interpreted `"Nothing else"` as a turn-stop. Results:
- **Codex**: 39→42 (+3) — the stall was preventing Codex from demonstrating correct loop execution; fixed.
- **Opus**: 48→49 (+1) — marginal improvement; Opus wasn't severely stalling before.
- **Sonnet**: 47→46 (−1, noise) — no regression.
- **GPT-5.4**: 44→39 (−5) — dropped notably. See GPT-5.4 analysis below.

The fix successfully eliminated "Nothing else" stall language from all 4 model failure analyses — no model mentioned `"Nothing else"` as a stall risk in this run.

### GPT-5.4 drop (44→39) — caused by "same turn" ambiguity, not regression

GPT-5.4's 5-point drop traces to MFA Clarity (8→7) and to failure mode analysis that highlighted a **new ambiguity** introduced by the fix: `"immediately continue to B–E in the same turn"` could be read as "fire all B–E in one parallel tool-call batch" rather than "sequential batches within the same response."

GPT-5.4 specifically called out: `"Continue to B–E in the same turn conflicts with the fact those steps depend on prior query results."` This is structurally correct — step B (CREATE TABLE) can run before C's result is known, but step D/E depend on step C's continuation decision. The phrase "same turn" implies simultaneity which contradicts the sequential dependency.

Opus independently raised the same finding: `"The instruction should clarify that 'same turn' means 'same response to the user, multiple sequential tool-call batches.'"` Sonnet also noted that C requires a "multi-branch decision tree."

**This is a 3/4 model convergence finding.** The "same turn" phrase was amended in the final v0.10.11 spec to: `"same response (sequential tool-call batches, not one parallel batch)"`.

### MFA Clarity improves across the board (7.75→8.0 avg)

Despite GPT-5.4's lower self-rating, the dimension average improved slightly (7.75→8.0), driven by Opus's 8→9 improvement. Opus specifically praised the explicit first-batch definition and the negative guardrail at line 45. The `"Nothing else"` removal contributed positively to Opus's structural assessment.

### All models correctly read both the v0.10.10 Frigg gate fix AND the v0.10.11 stall fix

Every model that enumerated the Step 3a gate correctly included `AND passed = 1` (v0.10.10 fix), and no model listed "Nothing else" as an ambiguous phrase (v0.10.11 fix confirmed working). Both prior-version fixes are confirmed absorbed.

### Top remaining cross-model finding: MFA→Loop transition is textual, not structural

2/4 models (Opus, Sonnet) independently identified that the MFA→Loop transition — a single prose sentence ("MFA complete. New tasks → begin the Odin Loop at Step 0.") — is the most likely cause of the "loop not always starting" symptom. Between MFA completion (line 43) and Step 0's first action (line 64), there are ~20 lines of identity/methodology prose. Models may process those as follow-on instructions rather than context-to-absorb.

**Opus's structural recommendation**: add `report_intent('Starting Odin Loop')` at MFA step E as a forcing function, keeping the model in execution mode rather than yielding.

**Sonnet's recommendation**: add an immediate `report_intent` call at the top of Step 0 (before the loop-entry gate check) as a commitment token.

This is the **highest-priority finding for v0.10.12**.
