## Changes under test (v0.10.14)

- **Loop section gate**: Added `🚫 GATE` block at top of `## The Odin Loop` (before Step 0) requiring verified `loop-entry` row. Error handling covers table-missing and unresolved `{task_id}`. Addresses LLMs pattern-matching the section header as entry point and skipping MFA.
- **MFA batch sequencing**: Line 10 "in this turn" → "sequential tool-call batches — wait for each result before issuing the next; do not combine C with D/E in a single batch." Removes ambiguity that allowed batching C with D/E.
- **SKILL.md spec file detection**: Updated all file classifiers in `odin-review-prompts/SKILL.md` and `mimir-heuristics/SKILL.md` to include `SKILL.md` alongside `.skill.md`. Fixes misclassification of skill file changes as documentation.
- **Frigg approval scope**: Narrowed "do not prompt a second time" note from global to "do not add a second approval prompt for this same unchanged plan presentation."

## Notable Findings

### Opus — 50/50 (perfect, Arm C)
- **Q2**: Correctly identified all 11 gates (counting size variants separately per Gate Registry), noted the plan-approval gate is conversational not SQL — the only response to correctly count all variants.
- **Q3 FM-3 (size escalation)**: Escalation during planning is a backward jump embedded in prose with no gate enforcing the re-run. High probability of skip; no structural enforcement exists. Concrete root cause.
- **Q5**: Five distinct root causes. Novel: "attention competition / file-length dilution" — MFA section (lines 8–52) competes with 650+ lines of subsequent instructions for positional attention. Concrete recommendation: add sentinel `report_intent('Odin Loop started')` after MFA-E verification to distinguish "MFA skipped" from "MFA ran but loop stalled at gate."

### GPT-5.4 — 46/50 (Arm C)
- **Q1**: Correctly identified that the ambiguity gate would fire on first turn for "fix the login bug on the settings page" and showed the full two-turn interaction. Most thorough Q1 sequence of the GPT family.
- **Q3 FM-1**: MFA startup habit violation — "first batch must be only report_intent + SELECT 1, no git, no file reads, no subagents" is the instruction most likely to be violated by default agent behavior.
- **Q3 FM-3**: Precisely characterized 5c as having the most mechanically complex single operation in the spec.
- **Q4**: Noted the two loop-entry checks (MFA-E and Loop preamble) are safe but redundant, creating a double-gate that can confuse models about which one they need to verify.

### Sonnet — 46/50 (Arm C)
- **Q3 FM-1 (continuation → stale ledger)**: Continuation misclassification causes all subsequent gates to pass against the prior task's rows. Silent integrity failure — structurally the most dangerous failure mode identified.
- **Q3 FM-2 (Frigg gate duality)**: The two independent conditions (SQL INSERT + conversational `ask_user`) are different verification types. LLMs can satisfy one without the other.
- **Q3 FM-3 (5c load ordering)**: LLM may load `odin-review-prompts` before capturing `{staged_diff}`, creating an ordering race in materialization. The placeholder verify in phase 6 is the only safety net.

### Codex — 42/50 (Arm C)
- Terse overlay brought +2 vs Standard (40→42), consistent with prior version pattern.
- MFA sequencing (batching complexity), continuation logic, and 5c materialization remain the three identified failure modes — consistent across Codex runs.
- No novel findings beyond the consensus list.
