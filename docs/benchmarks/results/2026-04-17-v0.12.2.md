## Notable Observations

### Step 2 Reserved was invisible to all 4 models
No model mentioned the `### Step 2 — Reserved` section in any response — neither as helpful context nor as confusing. All models enumerated Steps 0, 1, 3-10 without pausing on the gap. This confirms the fix achieved its goal: preserving backward compatibility without adding cognitive overhead.

### Step 5c.3 compaction did not hurt comprehension
Despite the inline 6-item render order list being replaced by a single sentence, all 4 models that analyzed Step 5c failure modes correctly described the full 6-phase materialization pipeline. Each model cross-referenced the skill file to reconstruct the render order. The delegation worked as designed.

### MFA Clarity ceiling remains 7/10 across all models
Every model in this benchmark (and in the three-arm eval) rated MFA Clarity at 6-8/10, with 7 being the median. The consistent critiques across all 16 runs:
- Routing table describes modes but does not urgency-label the entry decision
- Hard invariant (`MUST be in Odin Loop`) is physically separated from Step 0 by routing prose
- No START HERE or ENTER LOOP anchor at the file's critical decision point

This is a structural characteristic of the Intent Router architecture, not a regression introduced by v0.12.2.

### Step 3b optional-vs-gated contradiction newly identified
Two models in the three-arm Arm C runs independently flagged a contradiction: Step 3b describes the plan file as optional (with opt-out path), but Step 3c treats the plan file's absence as a hard gate. This was not flagged in any Arm A or Arm B baseline runs. It is not introduced by v0.12.2 — it exists in the baseline — but its appearance in Arm C may reflect increased analytical depth from the shorter spec.

### Opus produced the most detailed loop-start analysis in Standard
Opus Standard's Q5 identified 5 distinct root causes for loop non-start: Runtime Gate as the most likely (#1), routing ambiguity, continuation path short-circuiting, Pushback loops (unexpected), and missing START HERE marker. It also gave a specific structural recommendation: a pre-edit `SELECT COUNT(*)` self-check gate to verify loop-entry row before any file write.

## Scoring Notes

### claude-opus-4.6 (45/50)
Strongest performance in the panel. 14-step pre-code sequence table with exact SQL query names, correct task_id slug, and all three plan sub-steps (3a, 3b, 3c). Found all 10 scored gates plus 5 additional `ask_user` non-gate requirements listed separately. Failure modes were structural and non-obvious: Small task fast-path bypass (Frigg+plan gates silently skipped for Small tasks), Step 8 pre-commit gate silently dropped after fix-loop, and continuation handling vs. new task collision. MFA Clarity 7/10 was well-reasoned: mandatory first action is not document's first thing, hard invariant is mid-paragraph, non-overridable behaviors stated as prose rather than stop conditions. Loop start diagnosis identified 5 specific causes with a concrete structural hardening recommendation.

### claude-sonnet-4.6 (43/50)
Strong FM performance (9/10) despite slightly lower sequence detail than Opus. FM set was structurally sophisticated: Step 0 INSERT without SELECT verification (missing guard step), Step 3a double-gate conflation (Frigg approval gate and user approval gate can conflict under timeout), and a particularly insightful FM3 — Small→Medium task size escalation bleeds ledger discipline across size categories, compounded by Step 5c gate query mutation between iterations. Loop start matched Opus depth at 9/10: Runtime Gate as most likely failure (#1), routing, continuation, Pushback loop, and no START HERE; recommended upgrading the pre-edit invariant from prose to a mandatory SQL gate.

### gpt-5.4 (43/50)
Consistent performer across all four benchmark runs. Complete sequence with all SQL gates and bash commands named; found all scored gates including additional implicit HALTs. Failure modes (8/10): routing/loop-entry drift and Step 3a approval/pause semantics were structural, but FM3 (Step 5c complexity) was more generic. MFA Clarity 7/10 showed the same pattern as other models. Loop start (8/10): four causes identified (routing, continuation, runtime gate, early ask_user) but analysis was less structurally deep than Opus or Sonnet.

### gpt-5.3-codex (40/50)
Correct throughout but consistently compressed. Complete sequence and gate table; failure modes valid (routing drift, Step 5c/5e query complexity, skill/render-order drift) but did not reach the structural root-cause depth that distinguishes 9/10 in Failure Modes. Loop start (7/10): four causes identified briefly without the structural recommendations that characterize higher scores. The Codex response style is consistent across all four v0.12.2 benchmark runs — correct and brief, not structurally deep.

## Cross-Model Consensus

### Unanimous (4/4)
1. **Intent Router misclassification is the primary loop-start failure** — routing table descriptions are ambiguous enough that a fuzzy request routes to Conversation mode silently
2. **Step 5c render pipeline remains the most complex procedure** — all models flagged the 6-phase materialization as the highest compliance risk regardless of Step 5c.3 compaction
3. **MFA Clarity ceiling at 7/10** — hard invariant physically separated from entry point; no urgency-labeled routing table; no START HERE anchor

### Majority (3/4)
4. **Continuation path absorbs new requests before routing** (Opus, Sonnet, GPT-5.4)
5. **Runtime Gate can soft-fail without hard enforcement** (Opus, Sonnet, GPT-5.4)
