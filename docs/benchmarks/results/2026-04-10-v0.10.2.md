## Analysis

### Anti-stall fix validated by benchmark models
The v0.10.2 changes were motivated by a real-world bug where Odin stalled after the start signal on Medium tasks. Opus's Failure Mode #2 independently identifies "stopping at the start signal" as the most likely failure mode in the exception-list cognitive-load analysis, and explicitly notes the v0.10.2 anti-stall language "directly addresses this." GPT-5.4's Loop Start analysis also identifies "model pauses after start signal" as a cause. This is external validation — the models reading the instruction file see the problem the fix targets and confirm the fix is readable and actionable.

### New progress signals absorbed cleanly
Three of four models (Opus, Sonnet, GPT-5.4) explicitly include the new Frigg signal (`🔮 Plan drafted — sending to Frigg ({frigg_model}) for cross-model review...`) in their step sequences, correctly placed after model selection and before Frigg launch. No model was confused by the addition. Codex's compressed output omits it but correctly enumerates all steps in order. The reviewer signal (Step 5c) was not surfaced in step sequences — expected, as the benchmark question asks about pre-code actions and 5c is post-implementation.

### Sonnet ties Opus at 49 (new Sonnet all-time high)
Sonnet's improvement is driven by a stronger MFA Clarity analysis — 5 specific structural concerns (vs 4 in v0.9.12) with more detailed section-proximity observations. The "burden of proof swap" proposal in Loop Start (assume new task unless ledger proves otherwise) is the most architecturally novel fix proposal from any non-Opus model to date. Both Claude models now score 49 — the instruction file is near-perfectly comprehensible for Anthropic's model family.

### Codex dip (-2)
Codex dropped from 46 to 44, with less detailed Failure Mode (9→8) and Loop Start (9→8) analysis. The response was notably compressed — all sections present but with surface-level reasoning. This appears to be natural response variation: Codex correctly identified all steps, all gates, and all v0.10.2 additions in Sequence and Gates (both 10). The analytical dimensions (Failure Modes, Loop Start) are where compression hurts most. Codex has shown ±3 variance across benchmark versions (range: 37-47), making this result within its normal band.

### Convergent novel insights (2+ models)
- **Start signal as stall point** (Opus + GPT-5.4): Both independently identify the start signal as a likely pause point, validating the real-world bug report and the anti-stall fix.
- **Silent steps leave no audit trail** (Sonnet + Opus): Both note that Steps 0b-2 don't INSERT ledger rows, so skipping them is undetectable. Different framing: Sonnet calls it a "hallucination trap" (no feedback loop), Opus calls it a "cognitive load" problem (too many exception modes).

### Single-model novel observations worth investigating
- **Sonnet:** "Swap the burden of proof" for new-task classification — assume new task unless ledger proves loop is active. This would architecturally close the continuation-misclassification vulnerability.
- **Opus:** "Every code-change task" in the preamble excludes Investigation by its wording — an LLM might reason "Investigation isn't code-change → MFA not needed." Proposes "Every task — code change or investigation —".
- **Opus:** Architectural enforcement proposal — a tool-level guard that rejects `edit`/`create` calls without a `loop-entry` row. This is the strongest "beyond instruction text" remediation suggested in any benchmark.

### MFA Clarity plateau holds
All 4 models score 8-9 on MFA Clarity for the fourth consecutive version. The remaining gap is structural — forward references, "do not read further" being unenforceable, and competing directives. Opus and Sonnet both provide actionable suggestions (move MFA before character description, compress MFA to survive compaction) but neither can break through the 9 ceiling without architectural enforcement.
