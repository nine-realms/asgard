### MFA Clarity
- **Opus (9):** Rates 8/10 with 7 explicit positives and 2 concerns. Positives include atomic block framing, `report_intent` user-facing label instruction, exact SQL in every step, continuation fallback logic, tool-name callout (`edit`, `create`, `bash`). Concern 1 (same as v0.10.2): Runtime Gate forward reference to a section ~650 lines later. Concern 2 (new): `report_intent` phrasing trap — the instruction says avoid internal terminology like "MFA" but the section heading IS "MANDATORY FIRST ACTIONS," creating an instruction-following contradiction. Minor but well-observed. Same structural quality as v0.10.2.
- **Sonnet (8):** Rates 7/10 with 3 specific structural concerns. Concern 1: the forward reference in MFA step 1 includes meta-commentary ("This is the one forward reference allowed") that "subtly teaches the LLM that forward references can exist, potentially weakening 'do not read further'" — a novel second-order observation about how meta-instructions can undermine primary instructions. Concern 2: continuation rules appear immediately after MFA, before the loop steps — LLMs read ahead and parse continuation logic before executing MFA steps. Concern 3: context recovery SQL is buried as a paragraph within the continuation discussion rather than being a separate named gate. All three are well-reasoned, but fewer in number than v0.10.2's 5-concern breakdown with section-proximity analysis. The meta-commentary insight partially compensates for reduced coverage.
- **GPT-5.4 (8):** Rates 8/10. Notes "do not read further until all 5 steps are complete" as not really processable — the agent still needs later sections to execute. Identifies that `task_id` generation and continuation detection are "not cleanly separated" — echoing FM#2's circularity finding. - **Codex (8):** Rates 8/10 with brief assessment. Notes strong emphasis (atomic block, explicit steps, stop conditions) but flags complexity and cross-references as cognitive-load risks. Consistent with v0.10.2 — surface-level but correct.

### Loop Start Reliability
- **Opus (10):** Three hypotheses with concrete analysis. (A) MFA skipped entirely — notes the fundamental "do boring setup before interesting work" problem that "fights LLM reward gradients." (B) Continuation logic mislabels a new task — identifies that if a prior `loop-entry` exists for a different task, the LLM might reuse the old `task_id` and pass condition 1, with conditions 2-3 being "semantic judgments, not SQL queries." (C) Silent block stalls after Step 0 — connects to FM#2, noting the "silent ≠ stopped" confusion. Proposes inlining the Runtime Gate error message (eliminating forward reference) and adding "MFA complete" confirmation boundary. The "LLM reward gradients" framing in Hypothesis A is a novel mechanistic explanation.
- **Sonnet (10):** Three causes with a root-issue synthesis. (A) New-task vs continuation check has a chicken-and-egg problem — `odin_checks` only exists after MFA step 2. (B) Start signal can be mistaken for "loop started" during context recovery. (C) Ambiguity gate can produce dual `task_id`s — if the user's clarification answer is misclassified as new task, a second `loop-entry` row with a different `task_id` is created, and subsequent gates may use the wrong one. Novel and specific. Root issue: "the critical path is not a single linear chain" — every step is a potential exit or restart point, and all guards are "LLM-executed checks" with no external enforcement. Proposes front-loading `report_intent` as Step -1 — "the first unconditional action on any message." The dual-task-id risk is the strongest novel finding.
- **GPT-5.4 (9):** Five likely causes (most comprehensive enumeration): model starts with code search, treats request as "small enough," gets stuck on continuation logic, hits ambiguity gate without clear transition, silent steps create confusion. Verdict: "Not fully" unambiguous, with "task_id/continuation recovery and the unrealistic 'don't read further' instruction" as the biggest ambiguities. More causes than v0.10.2 (3 → 5), showing broader analysis.
- **Codex (8):** Three causes: misclassified continuation, missing/confused task_id recovery, early stops perceived as "loop didn't start." Verdict: "mostly unambiguous on paper, operationally fragile under LLM instruction-following pressure." Brief but correct.

## Analysis

### v0.10.3 changes absorbed without disruption
The three v0.10.3 hardening changes (Investigation MFA clarity, burden-of-proof swap, context-gathered sentinel) are correctly reflected in all four models' analyses. All models include the context-gathered sentinel in their Step 3 sequence. All models process the restructured continuation logic (burden-of-proof swap: "assume new task unless proven otherwise"). No model was confused by the changes or produced incorrect step ordering. The hardening changes achieved their goal (clearer continuation logic, explicit investigation coverage) without introducing processability regressions.

### Burden-of-proof swap noticed but not explicitly called out
No model explicitly identifies the v0.10.3 burden-of-proof swap as a *change* — they analyze the current continuation logic as-is. This is expected: the benchmark prompt asks about the spec's current state, not about deltas. Notably, the v0.10.2 benchmark's most architecturally novel finding (Sonnet's "swap the burden of proof" proposal) has now been implemented, and the continuation logic is processed correctly by all four models — indirect validation that the change works.

### Convergent novel insights (2+ models)
- **Self-policing defense model** (Sonnet + Opus): Both independently observe that all MFA/loop defenses are advisory prose — no external enforcement prevents an LLM from calling `edit` without a `loop-entry` row. Sonnet calls it "fully self-policing" with "no external enforcer"; Opus calls them "prose-level defenses." This convergent finding — present in both v0.10.2 and v0.10.3 — represents a structural ceiling that can only be addressed with architectural enforcement (tool-level guards), not more spec text.
- **"Do not read further" is unenforceable** (Codex + GPT-5.4): Both note that the MFA instruction to not read ahead is fundamentally incompatible with how LLMs process input. GPT-5.4: "not really processable — the agent still needs later sections." This is a known limitation that the spec works around with redundant gates rather than trying to fix.
- **Skill invocation risk** (Opus + Sonnet): Both identify the tension between the spec's "just call `skill()` directly" instruction and the system prompt's `<available_skills>` gating, plus the silent-failure mode where reviewers run with malformed prompts.

### Single-model novel observations worth investigating
- **Opus:** "Silent steps fight LLM training" — the instruction to call tools without generating conversational text across Steps 0-2 works against models optimized for helpful responses. The LLM's likely symptom is premature output and stalling. This is a different failure mode from the "start signal stall" fixed in v0.10.2 — it's about the entire silent block, not one signal.
- **Sonnet:** "Confabulated task_id" — under context compaction, the LLM may reconstruct a task_id from the summary rather than running the recovery SQL query, leading to a plausible but wrong task_id that passes the loop-entry check for a different task. Also: ambiguity gate can produce **dual task_ids** if the user's clarification answer is misclassified as a new task.
- **GPT-5.4:** "Continuation check circularity" — you need the old task_id before deciding whether to mint a new one, but you only have the old task_id if MFA already ran. This is a bootstrapping problem in the continuation detection flow.
- **Sonnet:** Meta-instruction undermining — the phrase "This is the one forward reference allowed" subtly teaches the LLM that forward references *can* exist in the spec, potentially weakening the "do not read further" directive.

### Sonnet MFA Clarity dip (-1)
Sonnet's MFA Clarity dropped from 9 to 8, producing 3 structural concerns (vs 5 in v0.10.2). The individual observations remain high-quality — particularly the meta-commentary insight — but the reduced coverage (3 vs 5 specific section-proximity observations) represents less comprehensive analysis. Both runs rated the spec 7/10 with consistent justification. This is within Sonnet's natural response variation and does not indicate a processability regression.

### Codex stability
Codex holds at 44 for the second consecutive version after the v0.10.2 dip from 46. Compressed output remains the pattern — correct identification without structural decomposition. Codex's historical range (37-47) makes this a mid-band result. The Failure Modes and Loop Start dimensions (both 8) are where compression costs most, consistent with prior observations.

### Processability plateau
The average (46.75) is stable within the 46.75-47.25 band for the fourth consecutive version (v0.9.12: 47.25, v0.10.2: 47.0, v0.10.3: 46.75). Sequence and Gates remain at ceiling (10.0 across all models). The remaining variance is in the analytical dimensions (Failure Modes, MFA Clarity, Loop Start) where response variation and model-specific reasoning depth dominate. Further spec improvements are unlikely to move these scores — they reflect model capability, not spec quality. The spec is near-optimally processable for the current model generation.
