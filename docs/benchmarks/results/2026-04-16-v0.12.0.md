**⚠️ This comparison is not apples-to-apples.** The v0.11.3 benchmark used the standard prompt (no terse overlay) and was scored against the MFA-based architecture. The v0.12.0 benchmark used the terse overlay and was scored against the new Intent Router architecture. Key differences:

1. **Q4 rubric mismatch:** The simulation prompt asks about "MANDATORY FIRST ACTIONS" — a specific section heading that existed in v0.11.3 but was removed in v0.12.0. The v0.12.0 architecture distributes this logic across Intent Router + Step 0. Models correctly noted the absence and scored themselves lower on Q4, which is an accurate observation about the new structure, not a regression in clarity.

2. **Q5 failure mode shift:** v0.11.3 models diagnosed MFA Step C continuation complexity. v0.12.0 models diagnosed Intent Router misclassification risk. Both are valid — the failure surface moved, not worsened.

3. **Baseline version difference:** Arms A/B used v0.10.17 (main), not v0.11.3 (feature branch). The three-arm C−B delta (+0.33) is the meaningful comparison.

## Change Summary

v0.12.0 replaces the MFA ceremony + Phase 1/2 architecture with an Intent Router + 3-mode dispatch:

1. **Intent Router** — fail-closed dispatcher classifies every message as Conversation / Odin Loop / Ship before any work begins
2. **Conversation mode (default)** — no SQL, no ceremony, just respond as a senior engineer
3. **Ship mode** — lightweight commit/push/PR path with guards (git status, diff display, ask_user confirmation)
4. **Odin Loop** — full verification pipeline, only entered for file mutations
5. **Hard invariant preserved:** `edit`/`create`/repo-write requires a verified `loop-entry` row
6. **Steps 3-10 unchanged** — all verification gates, reviewer launches, and evidence bundle logic preserved
7. **789 → 590 lines** (~25% reduction)

## Notable Observations

### Intent Router was correctly identified by all models
All 3 Arm C models correctly identified the routing table, the 3 modes, and the hard invariant. No model tried to trace through MFA Step C continuation logic (which no longer exists). This validates that the new entry path is structurally clearer.

### "MANDATORY FIRST ACTIONS" heading absence was flagged
Sonnet 4.6 specifically noted that the word "MANDATORY" does not appear in v0.12.0 and the critical sequence is distributed across Intent Router, Step 0, and Runtime Gate sections. This is a genuine structural observation — the v0.12.0 architecture trades a single "do this first" block for routing logic that directs to the right first action per mode.

### Failure modes shifted from continuation to routing
v0.11.3's top failure mode (all 4 models agreed): MFA Step C continuation branching complexity. v0.12.0's top failure mode (all 3 models agreed): Intent Router misclassifying a code-change request as Conversation. This is a different (and arguably simpler) failure surface — routing is a single decision point, not a multi-branch state machine.

### GPT-5.4 caught the Step 2 numbering gap
GPT-5.4 noted that `odin-recall` SKILL.md says "proceed to Step 2" but the agent file has no Step 2 (Steps go 1→3). This is a valid observation — the step numbering gap was intentional to preserve cross-references, but the recall skill's reference is stale. **Action item:** Update recall skill step reference.

### Codex scores were consistent across all 3 arms
Codex 5.3 scored 37 in all three arms, suggesting it processes both architectures at the same depth. Its responses were correct but consistently briefer, with less structural analysis in Q3-Q5.

### No regression in gate identification
All models found the same gate set (loop-entry, Frigg, plan file, baseline, 5c reviewer gates, 5e evidence, 8 pre-commit). The gate structure is unchanged between versions, and models correctly traced them.

## Scoring Notes

### claude-sonnet-4.6 (Arm C: 44/50)
Strongest Arm C performance. Excellent Q1 with correct new step numbering (1a-1h), full gate table, and three genuinely insightful failure modes: small-task misclassification as the highest risk, loop-entry ceremonial skip tendency, and SQL ledger drift for partial compliance. Q4 gave a well-justified 6/10 self-rating noting the distributed routing logic — scored 9 on the rubric because the structural analysis was excellent even though the self-rating was low. Q5 correctly identified 4 root causes including the continuation handler's open-task query behavior.

### gpt-5.4 (Arm C: 42/50)
Consistent performer. Very detailed 16-step first-turn sequence with correct v0.12.0 structure. Caught the Step 2 numbering gap between recall skill and agent file — a genuine cross-reference issue. Q3 failure modes were good (router ambiguity, numbering drift, mandatory/optional tension). Q4 and Q5 were solid but less structurally deep than Sonnet's analysis.

### gpt-5.3-codex (Arm C: 37/50)
Correct but compressed. All major steps and gates identified, but with less sub-step detail and shallower failure mode analysis. Consistent baseline across all 3 arms — processes both architectures at the same depth level. Q3-Q5 responses were valid but brief, missing the structural root-cause depth that distinguishes 8+ scores.

### Measurement Note: Arm B Codex contamination
The Arm B Codex run referenced v0.12.0 concepts ("Intent Router", "Step 1a-1f") despite being pointed at the main branch agent file. This suggests possible file-read confusion. Scores were kept as-is since the response content was internally consistent, but this run should be treated with lower confidence.
