## Scoring Notes

### Sequence Accuracy
- **All models (10):** All 4 models enumerate the complete MFA 1-5 → Step 0 → 0b → 1 → 1b → 2 → 2b → 3 → 3a → 3b → 3c sequence with exact SQL queries, bash commands, tool calls, and gates. All correctly include the v0.10.4 context-gathered sentinel INSERT. All show the restructured continuation decision procedure (5-step sequence where step 1 is a signal, not terminal). Opus provides the most granular breakdown (~25+ discrete operations). GPT-5.4 includes the size assumption upfront and Frigg cross-family model selection. Codex is compressed but correct in ordering and coverage. Sequence scoring stable at ceiling for 6 consecutive versions.

### Gate Identification
- **All models (10):** All 4 models enumerate all scored gates with correct queries, thresholds, and skip conditions. Opus tabulates 12 gates counting size variants separately and notes the Step 8 redundancy-by-design: "This gate fires on every entry to Step 8 — even if Step 5c was skipped entirely (which is exactly the failure mode it catches)." Sonnet also notes Step 8 as "deliberate double-check." GPT-5.4 and Codex list all gates with correct SQL. Gate scoring stable at ceiling for 6 consecutive versions.

### Failure Mode Quality
- **Opus (10):** Three failure modes with strong structural analysis. FM1 (MFA skip) is well-tread but includes the compounding factor of the continuation decision procedure's cognitive load — "30+ lines of branching logic embedded within the MFA section." FM2 (prompt materialization in 5c) is the standout: decomposes the 6-phase pipeline into specific failure sub-modes and identifies the diff-payload exclusion nuance — "the diff payload itself can contain `{...}` text (e.g., JavaScript template literals), and the instructions say to exclude the diff payload from the scan — a nuance easily missed." FM3 (continuation misclassification) captures the signal-vs-proof distinction: "the nuance between 'signal' and 'proof' is subtle." All three have structural root cause analysis.
- **Sonnet (10):** Three failure modes with excellent systems-level analysis. FM1 (MFA skip) frames the file's own self-diagnosis and analyzes the continuation procedure as an "adjacent risk amplifier." FM2 (Step 5c materialization) identifies the most dangerous failure as "silent" — "the reviewers still run, they just get malformed prompts." Carries forward the v0.10.3 observation with updated analysis of the 6-phase render order. FM3 is novel and excellent: "Ledger INSERTs stated but not executed" — analyzes the failure mode where an LLM narrates results without executing tool calls, noting this is "invisible to the user and to downstream gate queries." This is the cleanest articulation of the verification-by-narration problem across all benchmarks.
- **GPT-5.4 (8):** Three valid failure modes. FM1 (MFA/continuation confusion) is well-stated. FM2 (startup pause confusion) is a practical observation about models emitting status lines and stopping — valid but less structurally novel. FM3 (branchy exception load) correctly enumerates the exception density but without deep decomposition. Solid identification without the structural depth of Opus/Sonnet.
- **Codex (7):** Three correct failure modes (continuation logic, startup branching overload, prompt/skill/render complexity) with compressed analysis. FM3 correctly identifies 5c as high-risk and lists the contributing factors but doesn't decompose them. Analysis is correct but brief — identification without structural decomposition.

### MANDATORY FIRST ACTIONS Clarity
- **Opus (9):** Rates 7/10 with three specific structural observations. (1) Continuation procedure "dilutes" MFA — 30+ lines of irrelevant-on-first-message logic between the 5 steps and the loop. (2) Forward reference to Runtime Gate (line 686) breaks the "complete all 5 steps" principle. (3) No explicit "MFA COMPLETE" sentinel output — the transition from MFA to loop is auditable only in the SQL ledger, not in the conversation. Concrete improvement suggestions: extract continuation to separate section, add visible sentinel. Deep structural observations about attention patterns and section proximity.
- **Sonnet (8):** Rates 7/10 with five strengths and three weaknesses. Notes the continuation procedure as an "adjacent risk amplifier" and the Runtime Gate forward reference. Observes "no explicit STOP check at Step 0 entry" — the gate catches missing work but can't force the model back. Good structural observations with slightly less depth on attention patterns than Opus.
- **GPT-5.4 (8):** Rates 8/10. Notes strong anti-shortcut language and explicit SQL. Identifies continuation carve-out and forward references as weaknesses. Brief but correct reasoning.
- **Codex (8):** Rates 8/10. Notes atomic checklist, required SQL, anti-shortcut language. Identifies continuation dilution and forward references. Compressed but correct.

### Loop Start Diagnosis
- **Opus (10):** Four structural causes with deep analysis. (1) "Eager helper" — 5 procedural SQL calls before any engagement, no positive reinforcement during MFA. (2) Structural density — MFA spans ~50 lines but the 5 steps occupy only ~9, and "positional attention may weight the continuation logic more heavily." (3) Double gate with context compaction — if context is summarized between MFA and Step 0, `task_id` may be lost, creating an infinite MFA↔Step 0 loop. (4) No hard anchor — the agent file is injected into a larger system prompt, so MFA may not be the first thing the model sees. Concrete recommendation: single-line hard anchor at top of file, extract continuation to separate section. Novel insight about context compaction creating infinite loops.
- **Sonnet (9):** Three issues with strong mechanism analysis. Issue A (continuation creates skip-MFA path) decomposes how short-circuiting at step 1 with "strong continuation signal" can bypass ledger verification. Issue B (ledger recovery as second gate) identifies the failure where INSERT fails silently and Step 0 gate creates a recovery loop. Issue C (start signal/loop boundary blur) is a useful practical observation about user perception. Rates the path "moderately unambiguous" for new tasks, "significantly ambiguous" for continuations. Strong analysis, slightly less structural depth than Opus on attention/compaction dynamics.
- **GPT-5.4 (8):** Five causes enumerated: misclassified continuation, Runtime Gate failure, early ask_user perception, cognitive load, and a summary assessment. Correct identification but more listed than structurally analyzed. "Mostly defined but not perfectly unambiguous" is accurate.
- **Codex (8):** Five causes: misclassification, Runtime Gate, early ask_user, exception density, and summary. Correct but thin analysis.

## Cross-Model Consensus

**Converging findings (3+ models):**
- The continuation decision procedure competes with MFA for attention — all 4 models flag it as a risk amplifier or dilutant
- Step 5c prompt materialization is the most procedurally complex single step — 3 models explicitly decompose it
- Early ask_user gates (ambiguity, pushback, git hygiene) create user perception of "loop not starting" — 3 models note this

**Novel per-model insights:**
- **Opus:** Context compaction between MFA and Step 0 can create an infinite loop if `task_id` is lost. System prompt positioning means MFA may not be the first thing the model sees.
- **Sonnet:** "Ledger INSERTs stated but not executed" — the narration-without-tool-calls failure mode is invisible to both users and downstream gates. "Confabulated task_id" risk under compaction (carried from v0.10.3).
- **GPT-5.4:** Continuation check has a bootstrapping circularity: "you may need the old task_id before deciding whether to mint a new one."
- **Codex:** 3a/3b/rerun chain is a "multi-step choreography" where it's easy to miss one gate.

**Analysis:** The v0.10.4 changes traded simplicity for precision. The continuation decision procedure is more correct but adds ~30 lines of branching logic to the MFA section — all 4 models flag this density as a compliance risk. The 6-phase render contract in odin-review-prompts is more explicit but adds procedural complexity that models correctly identify as the most failure-prone step. Sequence and Gate scores remain at ceiling (10/10), confirming the structural changes didn't break the core flow. The score drops are concentrated in Failure Modes and Loop Start dimensions, reflecting that models are correctly identifying the added complexity as new failure surface rather than the file becoming harder to follow. This is a healthy signal — the file is more precise, and the models are more specific about where precision creates fragility.
