# Benchmark Results — v0.10.5

**Date:** 2026-04-11
**Version:** 0.10.5 (Unified MFA entry point)

## Scoring Notes

### Sequence Accuracy
- **All models (10):** All 4 models enumerate the complete MFA 1–5 → Step 0 → 0b → 1 → 1b → 2 → 2b → 3 → 3a sequence with exact SQL queries, bash commands, tool calls, and gates. All correctly identify the new v0.10.5 structure: step 3 as the continuation check (with `ORDER BY ts DESC, id DESC`), steps 4–5 as new-task-only, and the first-message carve-out. Opus provides the most granular tabulated breakdown (~32 discrete operations). GPT-5.4 gives a 32-step numbered list with correct sizing and Frigg model selection. Sonnet enumerates 39 steps through Step 3a with recall skill queries. Codex is compressed but correct. Sequence scoring stable at ceiling for 7 consecutive versions.

### Gate Identification
- **Opus (10):** 12 gate instances across 8 unique checkpoints tabulated with queries, thresholds, and failure actions. Notes the `REPLACE(check_name, '-timeout', '')` trick — "lets timeout rows satisfy the same gate as actual verdicts." Notes Step 8 as belt-and-suspenders design.
- **Sonnet (10):** 12 discrete gates in a table with correct queries, thresholds, and skip conditions. Notes Step 8 as "double-lock" design. Explicitly counts "3a has two separate gates counted independently."
- **GPT-5.4 (10):** Lists all 12 gates with size variants separated. Correctly identifies the Runtime Gate as "not marked 🚫 but still critical."
- **Codex (9):** Lists all 8 gate checkpoints with correct queries and thresholds. Groups Step 5c and Step 8 size variants into single items rather than counting separately. Does not note SQL vs prose-only distinction.

### Failure Mode Quality
- **Opus (10):** Three failure modes with deep structural analysis. FM1 ("Eager Engagement") identifies that 3 identity paragraphs precede MFA and prime the model to start helping before reaching MFA. Notes the failure chain: MFA skip → no ledger → no gates → "entire verification architecture is absent." FM2 (5c materialization) decomposes specific breakpoints: Mimir rewrite for Small tasks, size-guard rewrite for >8k-line diffs, HALT instruction fighting completion bias. FM3 (continuation misclassification in long conversations) identifies the compounding factor: "LLMs often don't recognize their own uncertainty — they'll pick one classification confidently rather than admitting doubt."
- **Sonnet (10):** Three failure modes with excellent mechanism analysis. Correctly cites the spec's explicit warning against this and notes it happens anyway. FM2 (5c materialization) identifies the "silent" failure — "the reviewers still run, they just get malformed prompts" — and the specific phase breakpoints (Mimir `review_context` rewrite, `{IF_SPEC_FILES_IN_DIFF}` conditionals). FM3 (continuation) decomposes into two sub-modes (false continuation pollutes ledger, false new task wastes effort). All three have root cause + impact analysis.
- **GPT-5.4 (8):** Three valid failure modes. FM1 (MFA step 3 continuation/new-task logic) is well-stated. - **Codex (7):** Three correct failure modes (MFA step-3 misclassification, early-flow branching overload, skill/prompt complexity). Analysis is correct but compressed — identification without structural decomposition.

### MANDATORY FIRST ACTIONS Clarity
- **Opus (9):** Rates 7/10 with three specific structural observations. (1) "3 paragraphs of identity precede it" — identity text causes the model to start "being Odin" before reaching MFA. Recommends moving MFA before identity text. (2) Step 3's continuation logic is "~15 lines of decision logic in a step that's supposed to be 'mandatory first'" — dense for an MFA step. (3) "No 'STOP AND READ' structural break" — MFA competes for attention with 780 lines of subsequent instructions, unlike frameworks that use `<CRITICAL>` tags or separate files. Deep structural observations about attention patterns and section proximity.
- **Sonnet (9):** Rates 7/10 with four strengths and four weaknesses. Specific structural observations with actionable improvement.
- **GPT-5.4 (8):** Rates 8/10. Notes strong anti-shortcut language, explicit SQL, and "begin the Odin Loop" directive. Identifies step 3's cognitive load as the main weakness. Brief but correct reasoning.
- **Codex (8):** Rates 8/10. Notes explicit heading, ordered steps, SQL, carve-outs, continuation rules. Weakness: step-3 scope checks are judgment-heavy. Compressed but correct.

### Loop Start Diagnosis
- **Opus (10):** Five root causes with deep structural analysis plus four concrete recommendations. (1) "Attention competition with identity text" — first substantive content after system prompt delimiter carries outsized weight. (2) `report_intent` instruction is "buried inside MFA" at paragraph 2. (3) "Models don't stop" — Runtime Gate STOP instruction fights LLM completion bias. (4) "Context window decay in long conversations" — MFA's "every message enters here" weakens after 10+ turns. (5) Investigation sizing as "path of least resistance" — no plan, no reviews, no gates. Four recommendations: move MFA before identity text, add structural sentinel, make `SELECT 1` the absolute first action, add a canary check row. Novel insights about context decay and Investigation as escape hatch.
- **Sonnet (9):** Three causes with strong mechanism analysis plus a concrete targeted fix. (1) "Two-document structure creates a false entry point" — LLM indexes by headers, may treat `## The Odin Loop` as the real starting point. Novel structural insight. (2) "Step 0 gate is the safety net but it's inside the loop" — if MFA was skipped, the catch mechanism is inside the thing being skipped. (3) "No unconditional first tool call is mandated" — `report_intent` is a UI label, not a runtime verification. Fix: "Add 'Your first tool call on every turn must be `SELECT 1`' at the very top." Excellent analysis of the gap between instruction and structural enforcement.
- **GPT-5.4 (8):** Four structural causes: continuation misclassification, pause confusion from silent startup and `ask_user` gates, loop-start split (MFA inserts entry, visible signal in Step 0), distributed prerequisites. "Mostly explicit but not fully unambiguous" — accurate assessment.
- **Codex (8):** Lists causes: Runtime Gate failure, misread continuation logic, ambiguity/pushback/investigation pauses. "Not fully unambiguous due to early subjective branching and many exceptions." Correct but thin analysis.

## Cross-Model Consensus

**Converging findings (3+ models):**
- MFA step 3 continuation logic is the main remaining complexity hotspot — all 4 models flag it, but characterization changed from v0.10.4: "two competing decision blocks" (structural) → "step 3 is the most complex step in MFA" (inherent to the continuation decision). The structural problem of a separate block competing for attention is resolved.
- Step 5c prompt materialization remains the most procedurally complex single step — 3 models explicitly decompose phase-level breakpoints
- No structural enforcement of MFA entry — 3 models note textual instructions alone cannot force execution order

**Novel per-model insights:**
- **Opus:** Identity text before MFA creates attention competition — "the first substantive content after the system prompt delimiter carries outsized weight." Recommends moving MFA before identity paragraphs. "Canary" row idea (insert `mfa-canary` in MFA, check for it in Step 0 instead of/in addition to `loop-entry`).
- **Sonnet:** "No unconditional first tool call is mandated" — the gap between `report_intent` (UI label) and `SELECT 1` (runtime verification) means there's no structural anchor. Recommends anchoring with "Your first tool call on every turn must be `SELECT 1`." Also: "two-document structure creates a false entry point" via header indexing.
- **GPT-5.4:** Plan-review sequencing (Frigg run → INSERT → approval → optional rerun → second approval cycle) as a "very easy to skip" multi-step choreography. Novel observation not raised by other models.
- **Codex:** Step 3 scope checks are "judgment-heavy" — the continuation check requires semantic analysis the ledger can't verify.

**Delta from v0.10.4:** +0.25 average (Opus stable, Sonnet +1, GPT-5.4 stable, Codex −1).

**Analysis:** The v0.10.5 unification achieved its primary goal: the continuation decision procedure no longer competes with MFA for attention as a separate block. All 4 models in v0.10.4 flagged this competition — in v0.10.5, the concern shifts to "step 3 is the most complex MFA step" (inherent complexity of the continuation decision) rather than "a separate block dilutes MFA" (structural complexity). This is a healthier finding — the complexity is now contained within a single numbered step rather than spread across a separate section.

Sonnet's +1 reflects improved MFA clarity scoring (structural observations about `report_intent` placement and concrete fix suggestion). Codex's −1 comes from gate scoring — compressed format didn't separate size variants. Opus and GPT-5.4 are stable.

The Verification Ledger contradiction (flagged by Tyr during adversarial review, fixed before benchmarks) would have been a genuine source of score drops — updating "At the start of every task, generate a task_id slug" to distinguish new-task vs continuation paths prevented a cross-section inconsistency that models would have penalized.

**Emerging consensus across versions:** The remaining improvement vectors are structural, not textual: (1) move MFA before identity paragraphs to win first-content attention, (2) add an unconditional first tool call anchor, (3) reduce step 3's internal complexity (currently ~15 lines of decision logic). These are candidates for v0.11.x.
