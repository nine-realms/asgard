### Sonnet 4.6

**Sequence**: SELECT 1 → CREATE TABLE → task_id → Step 0 (scan instructions, boost, ambiguity gate, pushback gate) → 0b (3 git checks) → 1 (config scan) → 1b (5 recall queries with exact SQL) → 2 (3 searches) → 2b (signal) → 3 (draft) → 3a (Frigg on gpt-5.4, INSERT, gate check) → 3b (plan file, gate check) → 3c (baseline INSERTs, gate check) → Step 4

**Gates found**: All 8 (3a Frigg, 3b plan file, 3c baseline, 5c Medium, 5c Large, 5e evidence, 8 pre-commit Medium, 8 pre-commit Large). Noted 3c lacks explicit query.

**Failure modes**:
1. Loop doesn't start — helpfulness impulse overrides SELECT 1. Persona paragraph between CRITICAL warning and MFA diverts attention.
2. Step 3c baseline skipped under context pressure — silent step between plan approval and implementation, weakest gate.
3. Step 5c skill failure rationalization — LLM may construct prompts itself instead of HALTing.

**MFA clarity**: 6/10. Structural fragmentation between MFA section and Runtime Gate section. No "loop start" marker. No "new task" definition. Persona paragraph competes for attention.

**Loop start diagnosis**: 3 specific causes — helpfulness impulse, "new task" ambiguity, no re-entry checkpoint. Path is "conceptually clear but operationally fragile."

### GPT-5.3-Codex

**Sequence**: SELECT 1 → CREATE TABLE → task_id → Step 0 (instructions, ambiguity, pushback) → 0b (git hygiene) → 1 (tooling) → 1b (recall) → 2 (survey) → 3 → 3a (Frigg, INSERT, gate) → 3b → 3c (baseline) → Step 4

**Gates found**: 6 explicit (3a, 3b, 3c, 5c, 5e, 8). Correct queries and thresholds.

**MFA clarity**: 8/10. Explicit and front-loaded. Risk: later rules can distract before first actions execute.

### GPT-5.4

**Sequence**: SELECT 1 → CREATE TABLE → task_id → Step 0 (report_intent, scan 4 instruction files, boost, size as Medium, ambiguity gate, pushback gate) → 0b (3 git commands with pushback logic) → 1 (config file scan, cache results) → 1b (recall: 4 session_store queries) → 2 (survey: reusable code, patterns, blast radius) → 2b (progress signal) → 3 (draft silently) → 3a (Frigg on gpt-5.4, INSERT verdict, gate query) → present plan → 3b (mkdir + write plan, gate test -s) → 3c (IDE diagnostics + build + test with baseline INSERTs, gate check) → Step 4

**Gates found**: All 7 distinct gates with exact queries. Noted 3c uses prose-described gate not a formal query. Added Runtime Gate as a non-emoji mandatory check.

**Failure modes**:
1. Pre-code gates too numerous — weaker models may jump to coding
2. `ask_user` pauses look like "loop never started"
3. Step 5c highly brittle — skill load, placeholder expansion, parallel launch, INSERT verdicts, timeout handling

**MFA clarity**: 8/10. Strong positioning but: only sql is smoke-tested; control flow is spread across many later sections.

**Loop start diagnosis**: Conceptually clear, operationally fragile. Runtime Gate stops wrong runtime. Silent steps = no output. Early ask_user = paused progress. Medium/Large requires many gates before code. Rated 6.5/10 reliable for average LLM.

## Fixes Applied After This Benchmark

Based on simulation findings, these changes were made to `agents/odin.agent.md` (716 → 725 lines):

| Fix | Convergence | What changed |
|-----|-------------|-------------|
| Explicit SELECT for 3c baseline gate | 2/3 models | Replaced prose "if you have zero rows" with `SELECT COUNT(*) FROM odin_checks WHERE task_id = '{task_id}' AND phase = 'baseline';` |
| 🔁 Loop start marker | 2/3 models | Added `🔁 ODIN LOOP STARTS HERE — MANDATORY FIRST ACTIONS (above) must be complete` divider before Step 0 |
| "New task" definition | 1/3 models (Sonnet) | Added paragraph after MFA list defining new task vs continuation |

**Not applied:**
- Collapse Runtime Gate into MFA (1/3 models) — inlining the 20-line error message would push MFA steps 2-4 below the fold, likely making attention worse

**Next benchmark should verify these fixes raised scores, especially dimensions 4 (MFA clarity) and 5 (loop start diagnosis).**
