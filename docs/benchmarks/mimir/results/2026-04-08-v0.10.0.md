## Comparative Review Analysis

### Sonnet vs GPT-5.4 on real diff

Both models produced well-structured reviews with correct walkthrough format, risk assessment, and effort scoring.

**Key differences:**
- **Effort scoring:** Sonnet scored 3 (Moderate), GPT-5.4 scored 4 (Heavy). GPT-5.4's higher score is arguably more appropriate for a 10-file, 494-insertion behavioral spec change.
- **Finding focus:** The models found entirely different issues — Sonnet caught cross-file model inconsistencies (agent file vs review-prompts skill), while GPT-5.4 caught a routing logic gap (HTML file classification). This diversity is ideal panel behavior.
- **Both findings are real:** All 3 findings were confirmed as legitimate issues worth fixing.

### Model swap validation

GPT-5.4 ties Sonnet at 45/50 on instruction comprehension and finds different (but equally valid) issues in comparative review. This validates the v0.10.0 model swap from `claude-sonnet-4.6` → `gpt-5.4` as Odin-spawned primary: no quality loss, different perspective, cost savings.

### Codex gap

gpt-5.3-codex at 35/50 confirms it's not suitable for Mimir's multi-pass synthesis — compressed responses miss structural nuances in failure analysis and CCA integration. Codex excels at pattern matching (good for Tyr) but lacks the analytical depth Mimir requires.

### Opus ceiling

claude-opus-4.6 at 49/50 found a novel coverage gap (Medium-task confidence filter) that no other model identified, plus a severity hesitancy bias and a context fragmentation timing tension. The config override mechanism (`mimir-model: claude-opus-4.6`) is validated for teams wanting maximum review quality.

## Convergent Novel Insights

Findings independently identified by 2+ models (high-confidence issues):
- **Medium-task confidence gap** (Opus + Sonnet): Panel confidence filter suppresses Medium-confidence surface bugs, but in Medium tasks there's no Heimdall/Thor/Loki to catch them. Tyr doesn't cover bugs.
- **"Obvious" is undefined** (Sonnet + GPT-5.4): Large-task lane deference says "avoid duplicating obvious single-file bugs" — load-bearing qualifier with no definition.
- **CCA skill load failure is silent** (Opus + Sonnet): No output field indicates whether the heuristic library loaded. Fallback to first-principles is a dramatic quality cliff with no signal to the developer.

Single-model novel observations:
- **Opus:** Severity hesitancy bias (prefer 🟡 over 🔴 creates systematic under-reporting); context fragmentation (CCA heuristics load after Pass 2 but some are most useful during Pass 2)
- **Sonnet:** Pruning rule is an anti-pattern (no version pinning, drift risk); dynamic analysis in skill instead of agent is a double loss on failure
- **GPT-5.4:** Diff Rule vs omission mandate tension; instruction sprawl creating checklist overload
